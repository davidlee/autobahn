## 1. Summary & Context

- **Brief**: [brief.md](../../brief.md) — boundary API design and architecture for `autobahn`
- **Implementation Plan**: [IP-001](./IP-001.md)
- **Design Revision**: [DR-001](./DR-001.md)
- **Change Drivers**: Greenfield repository — no prior code, specs, or governance artefacts exist

## 2. Motivation

- `spec-driver` owns workflow semantics but has no runtime orchestration layer
- Orchestration concerns (session lifecycle, harness adapters, sandbox adapters) are currently absent or ad-hoc
- `autobahn` establishes a clean boundary: `spec-driver` = workflow authority, `autobahn` = runtime substrate
- First-iteration scope: prove the architecture with a narrow vertical slice (reviewer flow)

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Functional Python package (`autobahn`) with hatch build system, matching spec-driver conventions
  - Typed core models (pydantic) for workflow context, session state, runtime policy, transition plans
  - Public API surface: check_prerequisites, load_context, transition_from_handoff, spawn/observe/terminate, reconcile (review ops gated on spec-driver CLI)
  - Adapter protocol definitions for harness and session backends (sandbox is policy/config, not protocol)
  - Artifact integration layer consuming spec-driver workflow files (YAML/TOML)
  - One concrete harness adapter (Claude Code) as vertical slice proof
  - Test infrastructure with meaningful coverage of the vertical slice
- **Operational Constraints**: Library-first — no CLI, daemon, or network transport required
- **Dependencies**:
  - Soft: spec-driver DE-107 (pydantic models) — design for future alignment, don't block
  - Soft: spec-driver review CLI commands (verb-noun) — needed for end-to-end vertical slice; tests use fixtures until available

## 4. Out of Scope

- CLI wrapper, daemon/service mode, TUI
- Multiple harness adapters beyond the first (Claude Code)
- Sandbox protocols or enforcement (sandbox is caller/nix/harness responsibility)
- Network transport, multi-client coordination, distributed execution
- Advanced subagent topology
- Transcript-native memory or prompt construction as primary abstraction

## 5. Approach Overview

- **Package Layout**: `autobahn/` flat package at repo root (mirrors spec-driver)
  - `api/` — public entrypoints
  - `models/` — pydantic models
  - `artifacts/` — spec-driver file consumption
  - `runtime/` — supervisor/session orchestration
  - `adapters/harness/`, `adapters/session/` — pluggable backends
  - structured logging for observation (event subsystem deferred)
- **Key Changes**:
  - Fix and modernise pyproject.toml (hatch backend, correct package name, clean dependencies)
  - Establish package skeleton with `__init__.py` and public API module
  - Define adapter protocols (harness, session backend)
  - Implement artifact loading from spec-driver workflow files
  - Implement session lifecycle core
  - Wire vertical slice 1: session lifecycle (load → transition → spawn → observe → reconcile)
  - Vertical slice 2 (review flow) gated on spec-driver review CLI contract

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `uv run pytest` passes with meaningful test coverage of the vertical slice
  - `ruff check` and `ruff format --check` pass with zero warnings
  - Public API is importable: `from autobahn.api import ...`
  - Contract tests pass against spec-driver-exported YAML fixtures
  - Adapter protocols are implementable (at least one concrete adapter each)
  - Session lifecycle testable via subprocess backend (real process, no tmux required)
  - Real subprocess integration test included (not just mocks)

## 7. Risks & Mitigations

- **Risk**: Over-engineering adapter abstractions before real usage — _Likelihood_: medium — _Impact_: medium — _Mitigation_: Keep protocols minimal; add methods only when the vertical slice demands them
- **Risk**: Artifact contract drift if spec-driver workflow files change — _Likelihood_: low (short-term) — _Impact_: high — _Mitigation_: Contract tests against spec-driver-exported fixtures; track DE-107 for shared schemas
- **Risk**: Boundary creep — autobahn defining spec-driver semantics — _Likelihood_: medium — _Impact_: high — _Mitigation_: Review CLI and review state machine are spec-driver's to define; DR explicitly defers to spec-driver authority

## 8. Follow-ups & Tracking

- **Future Deltas**: Additional harness adapters (pi-mono, Gemini, Codex), CLI wrapper, daemon mode, observation event subsystem, resume/pause session lifecycle, OTel integration
- **Open Decisions**: Async stance for public API (sync-first decided for now, revisit after vertical slice)
