# DR-001 – Repository bootstrap and architecture foundation

## 1. Executive Summary

- **Delta**: [DE-001](./DE-001.md)
- **Status**: draft
- **Last Updated**: 2026-03-22
- **Synopsis**: Establish `autobahn` as a typed Python library providing runtime orchestration of agent sessions driven by spec-driver workflow artifacts, with clean boundary discipline, pluggable adapters, and artifact-first recovery.

## 2. Problem & Constraints

**Current behaviour:**

- spec-driver owns workflow semantics, artifact schemas, and agent configuration (skills, hooks, preboot)
- No runtime layer exists to orchestrate agent sessions against workflow state
- Agent sessions are launched manually; handoff/review coordination is human-driven
- Session lifecycle is ad-hoc — no typed API, no reconciliation against artifact state
- Loss of a session requires manual re-orientation from workflow files

**Drivers:**

- `brief.md` — architecture brief defining boundary APIs and responsibility split
- spec-driver's workflow orchestration files (state.yaml, handoff.current.yaml, review-index.yaml, review-findings.yaml, sessions.yaml) are well-specified but have no programmatic consumer
- spec-driver DE-107 investigating pydantic for model layer — potential future convergence point (blocked by DE-106)

**Constraints / Guardrails:**

- Library-first — no CLI, daemon, or network transport in v1
- Must not duplicate or compete with spec-driver's workflow semantics
- Must not own sandbox orchestration (nix flake / srt / harness-internal)
- Must not own agent bootstrap configuration (spec-driver's skills/hooks/preboot)
- Artifact state is always authoritative over runtime/session state
- Must work with multiple harnesses (pi-mono preferred, Claude Code, Codex, Gemini as baseline)
- Track spec-driver's Python/tooling conventions (hatch, >=3.12, ruff, 2-space, pytest-asyncio)

**Out of scope:**

- CLI, daemon, service, TUI
- Sandbox protocols or enforcement
- Agent bootstrap file management
- Transcript replay or prompt-native memory
- Multi-client coordination, distributed execution
- Advanced sub-agent topology (sketch seams only)

## 3. Architecture Intent

### Target outcomes

- A Python library (`autobahn`) providing typed, coarse-grained orchestration of agent sessions driven by spec-driver workflow artifacts
- Clean three-layer ownership model: spec-driver configures → nix/srt sandboxes → autobahn orchestrates
- Pluggable harness adapters with tiered capability (baseline protocol + optional typed extensions)
- Recoverable from any failure using only artifact state + adapter policy

### Guiding principles

1. **Artifact-first continuity.** Workflow artifacts are the only durable truth. Runtime state is derived, not primary. If autobahn crashes, the next invocation reads artifacts and reconstructs what to do.

2. **Thin protocols, rich adapters.** Core protocols are minimal (4 ops for session backend, 2-3 for harness baseline). Richer harness capabilities (pi-mono's context engineering, sub-agents) compose via optional Protocol extensions — never bolted onto the base.

3. **autobahn doesn't configure agents.** spec-driver installs skills, hooks, preboot, CLAUDE.md. autobahn's job starts when the agent binary is ready to run. It provides the *runtime prompt* ("here's what to do now") not the *static bootstrap* ("here's who you are").

4. **autobahn doesn't sandbox agents.** The caller provides an executable — `claude`, `jailed-claude`, `srt claude`, whatever. autobahn runs it. Sandbox policy metadata can travel in `RuntimePolicy` for logging/observation, but autobahn doesn't enforce it.

5. **Sync reads, async runs.** Artifact consumption is synchronous pure computation. Session lifecycle is async. The boundary is natural and follows the data.

### State transitions / lifecycle

autobahn consumes but does not own the spec-driver workflow state machine (7 states, 7 commands). It reads `state.yaml` to determine current state, reads `handoff.current.yaml` to compute what to do next, and writes only to `sessions.yaml` (runtime-owned).

```
spec-driver state machine (read-only to autobahn):

  planned → implementing → awaiting_handoff → reviewing
                ↑                                 │
                │              changes_requested ←─┘
                │                     │
                └─────────────────────┘
                                            → approved

  any state ←→ blocked (with previous_state restore)
```

autobahn's orchestration loop:

```
  load_context()              # read artifacts, build WorkflowContext
       │
  transition_from_handoff()   # compute TransitionPlan from handoff + state
       │
  prime_review()              # (if reviewer) build/refresh review bootstrap
       │
  spawn_role_session()        # launch agent via harness adapter + session backend
       │
  observe_session()           # async: watch liveness, collect metadata
       │
  verify_review_outcome()     # (if reviewer) validate agent-written findings
       │
  reconcile()                 # compare artifact state vs session state, report drift
```

### Module responsibility boundaries

| Module | Owns | Does not own |
|--------|------|--------------|
| `api/` | Public entrypoints, input validation, typed returns | Internal orchestration logic |
| `models/` | Pydantic models, enums, result types | Persistence, I/O |
| `artifacts/` | Reading/writing spec-driver workflow YAML files | Workflow semantics, state transitions |
| `runtime/` | Session lifecycle coordination, reconciliation | Harness/session specifics (delegates to adapters) |
| `adapters/harness/` | Per-agent launch spec, exit interpretation | Process management |
| `adapters/session/` | Process create/alive/terminate/metadata | Agent-specific behaviour |
| `observation/` | Structured events, emission | Decision-making based on events |
| `_internal/` | Shared utilities (YAML, paths) | Business logic |

### Coupling rules

- `api/` depends on `models/`, `artifacts/`, `runtime/`
- `runtime/` depends on `models/`, `adapters/` (via protocols)
- `artifacts/` depends on `models/`, `_internal/`
- `adapters/` depends on `models/` only — never on `api/`, `runtime/`, or each other
- `models/` depends on nothing (leaf)
- `_internal/` depends on nothing (leaf)
- `observation/` depends on `models/` only

## 7. Adapter Protocols

### Harness adapter (baseline)

```python
class HarnessAdapter(Protocol):
  @property
  def name(self) -> str:
    """Adapter identifier, e.g. 'claude_code', 'pi_mono'."""
    ...

  def launch_spec(
    self,
    *,
    work_dir: Path,
    initial_prompt: str | None,
    resume_session: str | None,
    policy: RuntimePolicy,
  ) -> LaunchSpec:
    """Produce command, args, env to start or resume an agent."""
    ...

  def parse_exit(self, exit_code: int) -> SessionOutcome:
    """Interpret agent exit into structured outcome."""
    ...
```

### Harness extensions (seams — not locked)

Optional Protocol classes that richer harnesses (pi-mono) can implement. The runtime checks via `isinstance()` and uses extended capabilities when available. Baseline harnesses implement only `HarnessAdapter`.

```python
class ContextEngineerable(Protocol):
  """Harness supports structured context injection."""
  def context_sections(
    self, context: WorkflowContext,
  ) -> dict[str, str]: ...

class SubAgentConfigurable(Protocol):
  """Harness supports declaring sub-agent topology."""
  def sub_agent_spec(
    self, roles: list[Role], policy: RuntimePolicy,
  ) -> dict[str, LaunchSpec]: ...
```

These are illustrative. The specific methods will emerge when building the pi-mono adapter.

### Session backend

```python
class SessionBackend(Protocol):
  async def create(
    self, session_id: str, launch: LaunchSpec,
  ) -> SessionHandle: ...

  async def is_alive(self, handle: SessionHandle) -> bool: ...

  async def terminate(self, handle: SessionHandle) -> None: ...

  async def get_metadata(
    self, handle: SessionHandle,
  ) -> SessionMetadata: ...
```

Four operations. Tmux and subprocess backends both implement this. Tmux-specific capabilities (attach, capture-pane) follow the same optional extension pattern — deferred.

### Composition

The runtime supervisor holds both adapters:

```python
class Supervisor:
  def __init__(self, harness: HarnessAdapter, backend: SessionBackend):
    self.harness = harness
    self.backend = backend

  async def spawn(self, context, plan, policy) -> SessionHandle:
    launch = self.harness.launch_spec(
      work_dir=policy.work_dir,
      initial_prompt=plan.initial_prompt,
      resume_session=plan.resume_session,
      policy=policy,
    )
    return await self.backend.create(
      session_id=f"{context.artifact_id}-{plan.target_role}",
      launch=launch,
    )
```

Harness says what to run, backend runs it. Supervisor coordinates.

### Session operations (async)

```python
async def spawn_role_session(
  context: WorkflowContext,
  plan: TransitionPlan,
  *,
  policy: RuntimePolicy,
  harness: HarnessAdapter,
  backend: SessionBackend,

  *,
  backend: SessionBackend,

  handle: SessionHandle,
  *,
  backend: SessionBackend,
) -> OperationResult[SessionOutcome]:
  """Signal stop, return final outcome."""

async def reconcile(
  context: WorkflowContext,
  *,
  policy: RuntimePolicy,

```

Adapters are caller-provided. No service locator, no registry, no global state.

## 9. Artifact Integration Contract

### Read surface

| File | Required | autobahn reads |
|------|----------|---------------|
| `state.yaml` | Yes | All fields |
| `handoff.current.yaml` | No | All fields |
| `review-index.yaml` | No | All fields |
| `review-findings.yaml` | No | All fields |
| `sessions.yaml` | No | All fields |

### Write surface

| File | autobahn writes | Ownership |
|------|----------------|-----------|
| `sessions.yaml` | Full file | **autobahn-owned** |
| `state.yaml` | **Never** | spec-driver |
| `handoff.current.yaml` | **Never** | spec-driver |
| `review-findings.yaml` | **Never** | spec-driver tooling (invoked by agent) |
| `review-index.yaml` | **Never** | spec-driver tooling |

autobahn's only write authority is `sessions.yaml`. All other workflow mutations flow through spec-driver's CLI tooling, invoked either by autobahn (for priming) or by the agent (for review operations).

### Schema versioning

autobahn validates against version 1 of each schema. Unknown schema or unsupported version → raise, never silently degrade.

```python
SUPPORTED_SCHEMAS = {
  "supekku.workflow.state": {1},
  "supekku.workflow.handoff": {1},
  "supekku.workflow.review-index": {1},
  "supekku.workflow.review-findings": {1},
  "supekku.workflow.sessions": {1},
}
```

### Write protocol

All writes (sessions.yaml) follow spec-driver's atomic pattern:
1. Build content as pydantic model
2. Validate against schema
3. Write to temp file in same directory
4. Atomic rename

## 10. spec-driver Review CLI Contract

autobahn expects spec-driver to provide structured review commands (exact shapes are spec-driver's to define — `verb noun` convention). Starting-point contract:

### Commands autobahn invokes directly

```
spec-driver prime review <artifact-id>
  # Refreshes review-index.yaml
  # Returns: bootstrap status
  # Exit 0: success, Exit 1: error

spec-driver show review-status <artifact-id> [--format yaml]
  # Returns: bootstrap status, round, review status, finding counts
  # Structured output
```

### Commands the agent invokes during session

```
spec-driver add review-finding <artifact-id> \
  --title "..." --summary "..." \
  --status open --kind blocking|non_blocking \
  [--files path1 path2]

spec-driver complete review <artifact-id> \
  --status approved|changes_requested
  # Guards: cannot approve with open blocking findings
```

autobahn does not call `add review-finding` or `complete review` — those are the agent's responsibility. autobahn verifies the outcome after the session ends.

Note: this CLI surface does not yet exist in spec-driver and represents a dependency for the vertical slice. Exact command shapes will be defined by spec-driver.

### Review state machine

Review has its own lifecycle distinct from the parent workflow state machine:

```
                    ┌─────────────────────────────┐
                    │                             │
                    ▼                             │
  ┌──────────┐  prime   ┌──────────┐  begin    ┌─┴───────────┐
  │   cold   │────────►│   warm   │────────►│ in_progress  │
  └──────────┘          └──────────┘          └──────────────┘
       ▲                     ▲                   │         │
       │                     │                   │         │
       │              ┌──────────┐               │         │
       │   reprime    │  stale   │◄─ invalidate──┘         │
       │              └──────────┘                         │
       │                                                   │
       │                         ┌─────────────────────────┘
       │                         │                │
       │                   approve         request_changes
       │                         │                │
       │                         ▼                ▼
       │                  ┌──────────┐   ┌───────────────────┐
       │                  │ approved │   │changes_requested  │
       │                  └──────────┘   └───────────────────┘
       │                                          │
       │                                    resubmit
       │                                          │
       └──────────────────────────────────────────┘
```

Definition of states, transitions, and guards is spec-driver's responsibility. autobahn consumes the current state to make orchestration decisions.

## 11. Sandbox Landscape (Position Statement)

autobahn deliberately does not own sandbox orchestration.

### Three sandbox layers exist in practice

**1. Harness-internal sandboxing**
- Claude Code uses bwrap / sandbox-exec internally
- Codex uses its own sandbox-exec integration
- Opaque to autobahn

**2. Environment-level sandboxing (nix flake / jail-nix)**
- `jailed-agents.nix` composes agent + jail profile + env policy
- Profiles control: network, persistence, git push, identity, postgres, subagent depth
- autobahn sees `jailed-claude` as an opaque executable

**3. Standalone sandbox wrappers (Anthropic srt)**
- `@anthropic-ai/sandbox-runtime` — npm CLI wrapping commands with filesystem + network restrictions
- Uses bwrap / sandbox-exec + proxy-based network filtering
- Usable as command prefix (`srt claude ...`), not a Python library
- Research preview, API may evolve

### autobahn's position

autobahn sits above all three layers. `RuntimePolicy.sandbox_profile` carries metadata for logging/observation. autobahn does not enforce, configure, or verify sandbox state.

Building a sandbox abstraction in autobahn would be a worse version of what already exists at each layer.

### Future consideration

If autobahn needs to verify sandbox state (e.g. "confirm sandboxed before production access"), that would be an observation concern — querying metadata about the running environment — not an orchestration concern.

```python
  DRIFT_DETECTED = "drift_detected"
  ARTIFACT_READ = "artifact_read"
  ARTIFACT_WRITTEN = "artifact_written"

class Event(BaseModel):
  kind: EventKind
  timestamp: datetime
  artifact_id: str
  detail: dict[str, Any]
```

v1 emitter: JSONL file or Python `logging`. Future: Unix socket for spec-driver TUI integration (spec-driver already has `artifact_event.py` doing similar).

**Future consideration:** OpenTelemetry. Codex has configurable OTel support; spans for session lifecycle and events for state transitions would integrate with existing observability infrastructure. Design the event model to be OTel-compatible without requiring OTel as a dependency in v1.

### Reconciliation

Reconciliation is observation + comparison, not mutation:

```python
class DriftKind(StrEnum):
  SESSION_MISSING = "session_missing"
  SESSION_ORPHANED = "session_orphaned"
  STATUS_MISMATCH = "status_mismatch"
  STALE_REVIEW = "stale_review"

class DriftEntry(BaseModel):
  kind: DriftKind
  role: Role | None
  expected: str
  actual: str
  detail: str

class ReconciliationReport(BaseModel):
  artifact_id: str
  timestamp: datetime
  drift: list[DriftEntry]
  healthy: bool
```

`reconcile()` returns this report. The caller decides action. Repair operations can be added as separate functions that accept a `ReconciliationReport`.

Session state is advisory; artifact state is authoritative. Reconciliation detects where runtime reality has diverged from what artifacts say should be true.

## 13. Vertical Slice — Reviewer Flow

### Scenario

A delta is in `awaiting_handoff` state. The handoff targets the `reviewer` role. autobahn loads context, computes transition, primes the reviewer, spawns a review session, and after completion, verifies the outcome.

### End-to-end flow

```python
# 1. Load
ctx = load_context(artifact_dir)

# 2. Plan
plan = transition_from_handoff(ctx)

# 3. Prime
bootstrap = prime_review(ctx, git_head=get_current_head(ctx.artifact_dir))

# 4. Spawn
result = await spawn_role_session(
  ctx, plan,
  policy=RuntimePolicy(harness="claude_code", session_backend="tmux", ...),
  harness=ClaudeCodeAdapter(),
  backend=TmuxBackend(),
)

# 5. Observe
async for metadata in observe_session(result.value, backend=backend):
  if not metadata.alive:
    break

# 6. Verify
outcome = verify_review_outcome(ctx)

# 7. Reconcile
report = await reconcile(ctx, policy=policy, backend=backend)
```

### Test strategy

**Unit tests (no external dependencies):**
- Model validation against sample YAML fixtures
- Artifact loader/writer against fixture files
- API functions with deterministic inputs
- Transition planning for each state/role combination

**Integration tests (mock adapters):**
- Supervisor spawn flow with mock harness + mock session backend
- Spawn/observe/terminate lifecycle
- Reconciliation drift detection against mock session state

**Fixture data:** Sample `workflow/` directories with realistic YAML files derived from spec-driver's schema definitions.

**No real tmux/agent in CI.** Tests prove orchestration logic with mock adapters. Real integration tests are manual/optional.

### Deliberately not exercised

- pi-mono adapter or harness extensions
- Subprocess session backend (tested separately)
- Sandbox anything
- Multi-session coordination
- Auto-repair from reconciliation
- Resume/pause lifecycle

## 14. Design Decisions Summary

| ID | Decision | Rationale |
|----|----------|-----------|
| DEC-001 | Own pydantic models, structurally compatible with spec-driver | Hard boundary now; converge when DE-107 delivers shared schemas. Timing-dependent. |
| DEC-002 | Sync API for artifact ops, async for session lifecycle | Follows the data — file reads are sync, process management is async. No unnecessary async tax. |
| DEC-003 | Sandbox is policy/config, not protocol | Nix flake, srt, and harness internals already handle it. autobahn would build a worse version. |
| DEC-004 | Three-layer ownership | spec-driver configures → nix/srt sandboxes → autobahn orchestrates. Clean responsibility split. |
| DEC-005 | Thin baseline harness protocol + optional typed extensions | Baseline works for all harnesses. pi-mono gets richer support via composable Protocol extensions. |
| DEC-006 | 4-op async session backend protocol | Minimal: create, is_alive, terminate, get_metadata. Attach/capture are tmux-specific extensions. |
| DEC-007 | Caller-injected adapters, no registry | Keeps library stateless, composable, testable. |
| DEC-008 | Structured mutations via spec-driver CLI, not agent prose | Schema authority stays with spec-driver. Agent calls deterministic commands, not interpreted YAML writing. |
| DEC-009 | Review state machine owned by spec-driver | Review lifecycle is workflow semantics, not runtime. autobahn consumes current state. |
