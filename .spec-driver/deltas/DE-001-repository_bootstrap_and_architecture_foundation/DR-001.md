# DR-001 – Repository bootstrap and architecture foundation

## 1. Executive Summary

- **Delta**: [DE-001](./DE-001.md)
- **Status**: draft
- **Last Updated**: 2026-03-22
- **Synopsis**: Establish `autobahn` as a typed Python library providing runtime orchestration of agent sessions driven by spec-driver workflow artifacts, with clean boundary discipline, pluggable adapters, and artifact-first recovery.

## 2. Problem & Constraints

**Current behaviour:**

- spec-driver owns workflow semantics, artifact schemas, and agent configuration (skills, hooks, preboot)
- No runtime layer exists to orchestrate agent sessions against workflow state
- Agent sessions are launched manually; handoff/review coordination is human-driven
- Session lifecycle is ad-hoc — no typed API, no reconciliation against artifact state
- Loss of a session requires manual re-orientation from workflow files

**Drivers:**

- `brief.md` — architecture brief defining boundary APIs and responsibility split
- spec-driver's workflow orchestration files (state.yaml, handoff.current.yaml, review-index.yaml, review-findings.yaml, sessions.yaml) are well-specified but have no programmatic consumer
- spec-driver DE-107 investigating pydantic for model layer — potential future convergence point (blocked by DE-106)

**Constraints / Guardrails:**

- Library-first — no CLI, daemon, or network transport in v1
- Must not duplicate or compete with spec-driver's workflow semantics
- Must not own sandbox orchestration (nix flake / srt / harness-internal)
- Must not own agent bootstrap configuration (spec-driver's skills/hooks/preboot)
- Must not define spec-driver's CLI contracts — autobahn depends on them, does not author them
- Artifact state is always authoritative over runtime/session state
- Must work with multiple harnesses (pi-mono preferred, Claude Code, Codex, Gemini as baseline)
- Track spec-driver's Python/tooling conventions (hatch, >=3.12, ruff, 2-space, pytest-asyncio)

**Out of scope:**

- CLI, daemon, service, TUI
- Sandbox protocols or enforcement (see §11 for rationale)
- Agent bootstrap file management
- Transcript replay or prompt-native memory
- Multi-client coordination, distributed execution
- Advanced sub-agent topology (sketch seams only)
- Resume/pause session lifecycle (v1 is spawn/observe/terminate only)

**Relationship to brief.md:**

The brief calls for sandbox adapters as a full protocol layer. RE-001 formally revises that requirement based on analysis of the actual sandbox landscape (§11). The brief's sandbox adapter layer is replaced by opaque executable + policy metadata. All other brief requirements are preserved.

## 3. Architecture Intent

### Target outcomes

- A Python library (`autobahn`) providing typed, coarse-grained orchestration of agent sessions driven by spec-driver workflow artifacts
- Clean three-layer ownership model: spec-driver configures → nix/srt sandboxes → autobahn orchestrates
- Pluggable harness adapters with tiered capability (baseline protocol + optional typed extensions)
- Restartable from any failure using artifact state + adapter policy (v1 proves restart/reconstruction, not mid-session recovery)

### Guiding principles

1. **Artifact-first continuity.** Workflow artifacts are the only durable truth. Runtime state is derived, not primary. If autobahn crashes, the next invocation reads artifacts and reconstructs what to do.

2. **Thin protocols, rich adapters.** Core protocols are minimal (4 ops for session backend, 2-3 for harness baseline). Richer harness capabilities (pi-mono's context engineering, sub-agents) compose via optional Protocol extensions — never bolted onto the base.

3. **autobahn doesn't configure agents.** spec-driver installs skills, hooks, preboot, CLAUDE.md. autobahn's job starts when the agent binary is ready to run. It provides *structured references* to workflow state — spec-driver owns prompt content.

4. **autobahn doesn't sandbox agents.** The caller provides an executable — `claude`, `jailed-claude`, `srt claude`, whatever. autobahn runs it. Sandbox policy metadata can travel in `RuntimePolicy` for logging/observation, but autobahn doesn't enforce it.

5. **Sync reads, async runs.** Artifact consumption is synchronous pure computation. Session lifecycle is async. The boundary is natural and follows the data.

6. **Fail fast on missing prerequisites.** autobahn validates runtime preconditions (harness executable exists, backend available, required artifact files present) before attempting operations. Environment problems surface immediately, not mid-orchestration.

### State transitions / lifecycle

autobahn consumes but does not own the spec-driver workflow state machine. It reads `state.yaml` to determine current state, reads `handoff.current.yaml` to compute what to do next, and writes only to `sessions.yaml` (runtime-owned).

Authoritative reference: spec-driver `state_machine.py` (DR-102 §4), as of 2026-03-22. 7 states, 7 commands. DE-109 is actively formalising this — autobahn must track changes. The state machine is not reproduced here to avoid embedding a copy that will drift; see the source file for the transition table.

autobahn's orchestration loop (v1):

```
  load_context()              # read artifacts, build WorkflowContext
       │
  transition_from_handoff()   # compute TransitionPlan from handoff + state
       │
  spawn_role_session()        # launch agent via harness adapter + session backend
       │
  observe_session()           # async: watch liveness, collect metadata
       │
  terminate_session()         # (if needed) signal stop
       │
  reconcile()                 # compare artifact state vs session state, report drift
```

Review-specific operations (`prime_review`, `verify_review_outcome`) are gated on spec-driver's review CLI contract (OQ-001) and form a second vertical slice.

## 4. Three-Layer Ownership Model

```
┌─────────────────────────────────────────────────────────┐
│ spec-driver                                             │
│  • workflow semantics, artifact schemas                 │
│  • skills, hooks, preboot installation                  │
│  • review CLI contract (to be defined by spec-driver)   │
│  • review state machine definition                      │
│  • prompt content via templates and CLI                  │
├─────────────────────────────────────────────────────────┤
│ nix flake / srt / harness-internal                      │
│  • sandbox enforcement (filesystem, network, git)       │
│  • jail profiles (specDev, research, offline)           │
│  • agent binary wrapping                                │
├─────────────────────────────────────────────────────────┤
│ autobahn                                                │
│  • session lifecycle orchestration (spawn/observe/term) │
│  • artifact consumption + runtime coordination          │
│  • harness/session adapter management                   │
│  • reconciliation, drift detection                      │
│  • runtime precondition validation                      │
└─────────────────────────────────────────────────────────┘
```

autobahn receives an opaque executable and runs it. Whether that executable is `claude`, `jailed-claude`, or `srt claude` is the caller's decision.

**Environment preconditions:** Before orchestration, autobahn validates:
- Harness executable is resolvable on PATH (or at explicit path)
- Session backend is available (e.g. tmux binary exists, or subprocess is functional)
- Required artifact files exist and pass schema validation
- Workflow state is not terminal (approved) unless reconciliation-only

These checks fail fast with specific errors, not mid-operation.

## 5. Module Boundaries

Flat package at repo root (`autobahn/`), mirroring spec-driver convention. The file tree is intentionally not prescribed — module boundaries and coupling rules are the design constraints; implementers should organise files to serve them.

### Module responsibility boundaries

| Module | Owns | Does not own |
|--------|------|--------------|
| `api/` | Public entrypoints, input validation, typed returns | Internal orchestration logic |
| `models/` | Pydantic models, enums, result types, error taxonomy | Persistence, I/O |
| `artifacts/` | Reading/writing spec-driver workflow YAML files | Workflow semantics, state transitions |
| `runtime/` | Session lifecycle coordination, reconciliation | Harness/session specifics (delegates to adapters) |
| `adapters/harness/` | Per-agent launch spec, exit interpretation | Process management |
| `adapters/session/` | Process create/alive/terminate/metadata | Agent-specific behaviour |
| `_internal/` | Shared utilities (YAML, paths) | Business logic |

### Coupling rules

- `api/` depends on `models/`, `artifacts/`, `runtime/`
- `runtime/` depends on `models/`, `adapters/` (via protocols)
- `artifacts/` depends on `models/`, `_internal/`
- `adapters/` depends on `models/` only — never on `api/`, `runtime/`, or each other
- `models/` depends on nothing (leaf)
- `_internal/` depends on nothing (leaf)

### Artifact models (`models/artifacts.py`)

Provisional pydantic models for spec-driver's workflow YAML schemas. Schema authority resides in spec-driver. These models are validated by contract tests against authoritative fixtures.

**autobahn does not mirror the full schemas.** Models cover only the fields autobahn actually reads and depends on. Unknown fields are ignored (pydantic `model_config = ConfigDict(extra="ignore")`). This limits the coupling surface — spec-driver can add fields without breaking autobahn.

Fields autobahn depends on per file:

| File | Fields consumed |
|------|----------------|
| `state.yaml` | `artifact.id`, `artifact.kind`, `workflow.status`, `workflow.active_role`, `workflow.claimed_by`, `phase.id`, `phase.status`, `pointers.*`, `timestamps.updated` |
| `handoff.current.yaml` | `artifact.id`, `transition.from_role`, `transition.to_role`, `transition.status`, `phase.id`, `required_reading`, `next_activity.kind`, `open_items` |
| `review-index.yaml` | `review.bootstrap_status`, `staleness.cache_key.*` |
| `review-findings.yaml` | `review.round`, `review.status`, count of `blocking`/`non_blocking`/`resolved` |
| `sessions.yaml` | All fields (autobahn-owned, full model) |

**Note on phase sheet fields:** DE-106 (approved) establishes `plan`, `delta`, `objective`, `entrance_criteria`, `exit_criteria` as canonical phase frontmatter. These are not in the consumed fields table above because autobahn reads phase metadata from `state.yaml` (phase.id, phase.status), not from phase sheet files directly. If slice 1 needs to gate-check phase readiness (entrance criteria) or assemble reviewer context (objective), the read surface must expand to include phase sheets. This is a latent requirement to validate during implementation.

Models:
- `WorkflowStateFile` — stable subset of `supekku:workflow.state@v1`
- `HandoffFile` — stable subset of `supekku:workflow.handoff@v1`
- `ReviewIndexFile` — minimal: bootstrap status + staleness key
- `ReviewFindingsFile` — minimal: round, status, finding counts
- `SessionsFile` — full model (autobahn-owned)

### Error taxonomy (`models/errors.py`)

```python
class AutobahnError(Exception):
  """Base for all autobahn errors."""

# Artifact layer
class ArtifactParseError(AutobahnError):
  """YAML is malformed or unreadable."""

class ArtifactContractError(AutobahnError):
  """YAML is valid but violates schema contract
  (unknown schema, unsupported version, missing required fields)."""

# External tool layer
class ToolInvocationError(AutobahnError):
  """spec-driver CLI or other external tool failed to execute."""

class ToolContractError(AutobahnError):
  """External tool returned unexpected output format or
  exit 0 with invalid payload."""

# Backend layer
class BackendUnavailableError(AutobahnError):
  """Session backend prerequisites not met (tmux missing, etc.)."""

class BackendTransportError(AutobahnError):
  """Communication with backend failed mid-operation."""

# Policy layer
class PreconditionError(AutobahnError):
  """Runtime precondition not met (harness not found,
  terminal state, missing required files)."""

# Session layer
class SessionError(AutobahnError):
  """Session operation failed (already dead, not found, etc.)."""
```

**Convention:** These exceptions are for invariant violations and environment failures — bugs, incompatible environments, broken contracts. `OperationResult` is for expected operational outcomes — session ended unexpectedly, drift detected, review not yet complete. Each public API function documents which error classes it may raise and when it returns a failed `OperationResult` instead.

## 7. Adapter Protocols

### Harness adapter (baseline)

```python
class HarnessAdapter(Protocol):
  @property
  def name(self) -> str:
    """Adapter identifier, e.g. 'claude_code', 'pi_mono'."""
    ...

  def is_available(self, policy: RuntimePolicy) -> bool:
    """Check if this harness can run (executable exists, etc.).
    Used by check_prerequisites()."""
    ...

  def launch_spec(
    self,
    *,
    work_dir: Path,
    plan: TransitionPlan,
    policy: RuntimePolicy,
  ) -> LaunchSpec:
    """Produce command, args, env to start an agent.
    Harness builds launch args from plan's structured references
    (artifact_id, phase_id, handoff_path).
    Policy.harness_executable overrides adapter's default command."""
    ...

  def parse_exit(self, exit_code: int) -> SessionOutcome:
    """Interpret agent exit into structured outcome."""
    ...
```

### Harness extensions (seams — not locked)

Optional Protocol classes that richer harnesses (pi-mono) can implement. The runtime checks via `isinstance()` and uses extended capabilities when available. Baseline harnesses implement only `HarnessAdapter`.

```python
class ContextEngineerable(Protocol):
  """Harness supports structured context injection."""
  def context_sections(
    self, context: WorkflowContext,
  ) -> dict[str, str]: ...

class SubAgentConfigurable(Protocol):
  """Harness supports declaring sub-agent topology."""
  def sub_agent_spec(
    self, roles: list[Role], policy: RuntimePolicy,
  ) -> dict[str, LaunchSpec]: ...
```

These are illustrative seams. The specific methods will emerge when building the pi-mono adapter.

### Prompt/bootstrap seam

autobahn does not construct agent prompts. The contract between autobahn and the harness/bootstrap layer:

1. **autobahn provides:** `TransitionPlan` containing structured references — `artifact_id`, `phase_id`, `handoff_path`, `required_reading` (file paths)
2. **The harness adapter translates** these into launch arguments using harness-specific conventions (e.g. Claude Code's `-p` flag, pi-mono's context system)
3. **spec-driver's installed bootstrap** (CLAUDE.md, skills, preboot, hooks) is already on disk — spec-driver installed it. The agent reads it on boot.
4. **If richer runtime context is needed,** the agent calls a spec-driver command during its session that reads workflow artifacts. spec-driver owns the prompt rendering.

autobahn's TransitionPlan is a *pointer to work*, not *instructions for work*. The harness decides how to present that pointer to the agent. For Claude Code, this might be `claude -p "resume DE-001 phase IP-001.PHASE-03"`. For pi-mono, it might be structured context injection via the extension protocol. The details are harness-specific, not autobahn-specified.

### Session backend

```python
class SessionBackend(Protocol):
  def is_available(self) -> bool:
    """Check if this backend's prerequisites are met
    (e.g. tmux binary exists). Used by check_prerequisites()."""
    ...

  async def create(
    self, session_id: str, launch: LaunchSpec,
  ) -> SessionHandle: ...

  async def is_alive(self, handle: SessionHandle) -> bool: ...

  async def terminate(self, handle: SessionHandle) -> None: ...

  async def get_metadata(
    self, handle: SessionHandle,
  ) -> SessionMetadata: ...
```

Five operations: is_available (sync probe), create, is_alive, terminate, get_metadata. This is the v1 protocol — deliberately limited to spawn/observe/terminate.

**What v1 does not include:** resume, pause, attach, capture-pane. These are real requirements from the brief but require design work that the first slice doesn't need. They will be added as protocol extensions (same pattern as harness extensions) when the design for recoverable sessions is proven. Until then, session recovery works by re-spawning from artifact state, not by resuming a backend session.

### Composition

The runtime supervisor holds both adapters:

```python
class Supervisor:
  def __init__(self, harness: HarnessAdapter, backend: SessionBackend):
    self.harness = harness
    self.backend = backend

  async def spawn(self, context, plan, policy) -> SessionHandle:
    launch = self.harness.launch_spec(
      work_dir=policy.work_dir,
      plan=plan,
      policy=policy,
    )
    return await self.backend.create(
      session_id=f"{context.artifact_id}-{plan.target_role}",
      launch=launch,
    )
```

Harness says what to run (from plan's structured references), backend runs it. Supervisor coordinates.

Adapters are caller-provided. No service locator, no registry, no global state.

### Read surface

### sessions.yaml ownership semantics (provisional — OQ-005)

**This section describes autobahn's proposed write semantics. The authoritative ownership definition must be co-defined with spec-driver as the schema owner (OQ-005). autobahn must not treat these semantics as settled until that contract exists.**

Proposed preconditions:

- **Single-writer:** Only one autobahn process may orchestrate a given artifact at a time.
- **Detection:** Lockfile (`sessions.lock`) with PID and timestamp. Stale locks (PID dead) reclaimed.
- **Crash recovery:** Atomic rename prevents corruption. Stale locks reclaimed by next invocation.
- **Agent reads:** Agents may read `sessions.yaml` but must not write it.

Open questions for the cross-repo contract (OQ-005):
- Does spec-driver ever write to sessions.yaml?
- What are the interop rules if multiple tools observe/update session state?
- Should the lock mechanism be spec-driver-defined (consistent across workflow files)?

### Schema versioning

autobahn validates against version 1 of each schema. Unknown schema or unsupported version → `ArtifactContractError`, never silently degrade.

```python
SUPPORTED_SCHEMAS = {
  "supekku.workflow.state": {1},
  "supekku.workflow.handoff": {1},
  "supekku.workflow.review-index": {1},
  "supekku.workflow.review-findings": {1},
  "supekku.workflow.sessions": {1},
}
```

### Write protocol

All writes (sessions.yaml) follow spec-driver's atomic pattern:
1. Acquire lock (sessions.lock with PID)
2. Build content as pydantic model
3. Validate against schema
4. Write to temp file in same directory
5. Atomic rename
6. Release lock

### Contract test strategy

autobahn validates its models against representative YAML fixtures:

**Target state (when DE-110 lands):** spec-driver exports fixture YAML files generated from its own schema builders. autobahn's contract tests load these and validate that pydantic models parse them without error. On schema drift, contract tests break at CI time.

**Gap period (before DE-110):** autobahn hand-maintains fixtures in `tests/fixtures/` derived from spec-driver's existing test data and schema documentation. These are copies, not generated — they are a known drift risk. This is explicitly acknowledged as degraded testing posture.

**Risk:** Hand-maintained fixtures are the same hidden coupling that DEC-001 aims to avoid, just moved to test data. The mitigation is:
- Fixtures are small (one sample per file type)
- They are derived from spec-driver's documented schemas, not invented
- DE-110 replaces them with generated fixtures when it lands
- If DE-110 doesn't land, this remains a manual sync obligation

**Fixture location:** `tests/fixtures/` in autobahn. Each fixture carries a comment noting the spec-driver commit it was derived from.

### Schema version support policy

autobahn hard-fails on unsupported schema versions. This means a spec-driver upgrade that ships `@v2` of any workflow schema will break autobahn until updated. No forward-compatibility or graceful degradation is provided in v1.

**Future consideration:** If spec-driver adopts a schema evolution policy (e.g. v1+v2 coexistence period), autobahn should support reading both during the transition. This is unaddressed in v1 and should be co-designed when the first schema version bump occurs.

### Layout assumptions

autobahn assumes:
- Workflow files live in `{artifact_dir}/workflow/` (configurable via `workflow_dir` parameter)
- File names are fixed: `state.yaml`, `handoff.current.yaml`, `review-index.yaml`, `review-findings.yaml`, `sessions.yaml`
- Schema IDs follow the pattern `supekku.workflow.{name}`
- `phase_id` is derivable from `state.yaml` `phase.id` field

These assumptions are encoded in `_internal/paths.py` and validated by the contract tests. If spec-driver introduces an artifact locator/descriptor contract, autobahn can adopt it.

## 10. spec-driver Dependencies

autobahn depends on spec-driver for two categories of contract:

### Workflow artifact schemas (available now)

The 5 YAML schemas (`supekku:workflow.*@v1`) are defined and stable. autobahn consumes them via file reads. Contract tests validate compatibility.

**Convergence path:** DE-110 (spec-driver) scopes fixture exports for cross-repo contract testing. This is the actionable mechanism for schema convergence. DE-107 (pydantic migration) is internal to spec-driver — it does not create a public schema package or shared import surface. autobahn's models are fixture-validated copies, not future re-exports.

### Review CLI contract (not yet defined — OQ-001)

autobahn's review operations (`prime_review`, `verify_review_outcome`) depend on spec-driver providing structured review commands. The exact command shapes, output formats, and guard semantics are spec-driver's to define.

**What autobahn needs from this contract:**
- A command to prime/refresh review bootstrap state (reads handoff, writes review-index.yaml)
- A command to query current review status (structured output: bootstrap status, round, review status, finding counts)
- Commands for agents to record findings and complete reviews (spec-driver enforces guards like "cannot approve with open blocking findings")

**What autobahn does NOT define:**
- Command names, argument shapes, output schemas — those are spec-driver's verb-noun CLI design
- Review state machine states, transitions, guards — those are workflow semantics

Until this contract exists, review operations are untestable end-to-end. The first vertical slice (§13) is designed to work without it.

### Review state machine (not yet defined — OQ-003)

Review has its own lifecycle distinct from the parent workflow state machine — bootstrap lifecycle (cold → warm → stale) interleaved with judgment lifecycle (not_started → in_progress → approved/changes_requested). The formal definition of states, transitions, and guards belongs in spec-driver. autobahn will consume the current state to make orchestration decisions, once defined.

## 11. Sandbox Landscape (Position Statement)

**This section supersedes the brief's "Sandbox adapter layer" (§5.5 of brief.md).**

autobahn deliberately does not own sandbox orchestration. This is a design decision (DEC-003), not an omission.

### Three sandbox layers exist in practice

**1. Harness-internal sandboxing**
- Claude Code uses bwrap / sandbox-exec internally
- Codex uses its own sandbox-exec integration
- Opaque to autobahn

**2. Environment-level sandboxing (nix flake / jail-nix)**
- `jailed-agents.nix` composes agent + jail profile + env policy
- Profiles control: network, persistence, git push, identity, postgres, subagent depth
- autobahn sees `jailed-claude` as an opaque executable

**3. Standalone sandbox wrappers (Anthropic srt)**
- `@anthropic-ai/sandbox-runtime` — npm CLI wrapping commands with filesystem + network restrictions
- Uses bwrap / sandbox-exec + proxy-based network filtering
- Usable as command prefix (`srt claude ...`), not a Python library
- Research preview, API may evolve

### Why not a sandbox adapter protocol?

- The nix flake composes sandbox + agent at a higher level than autobahn can reach
- Harness-internal sandboxing is opaque by design
- srt is a CLI wrapper, not a composable Python protocol
- Building a sandbox abstraction in autobahn would be a worse version of what already exists at each layer

### autobahn's position

autobahn sits above all three layers. `RuntimePolicy.sandbox_profile` carries metadata for logging/observation. autobahn does not enforce, configure, or verify sandbox state.

### Future consideration

If autobahn needs to verify sandbox state (e.g. "confirm sandboxed before production access"), that would be an observation concern — querying environment metadata — not an orchestration concern.

## 12. Observation & Reconciliation

### Observation (v1: structured logging only)

v1 uses Python's `logging` module with structured log records. No custom event subsystem, no JSONL files, no Unix sockets. This is sufficient for debugging and can be consumed by any standard log handler.

```python
import logging
logger = logging.getLogger("autobahn")

# Example structured log entries:
logger.info("session_spawned", extra={
  "artifact_id": "DE-001",
  "role": "reviewer",
  "session_id": "DE-001-reviewer",
})
```

**v1 limitation: TUI-invisible sessions.** spec-driver's TUI (DE-053/054/059) consumes artifact events via JSONL + Unix socket (DE-052). autobahn v1 does not emit compatible artifact_event records. This means sessions managed by autobahn will not appear in the spec-driver TUI's real-time tracking. This is a known integration gap, not an oversight.

**Future considerations:**
- Artifact event emission compatible with spec-driver's event infrastructure (DE-052)
- OpenTelemetry spans for session lifecycle (Codex has configurable OTel support)
- Unix socket integration with spec-driver TUI
- Custom event types and JSONL emission

Artifact event compatibility is the highest priority follow-up for observation.

### Reconciliation

Reconciliation is observation + comparison, not mutation:

```python
class DriftKind(StrEnum):
  SESSION_MISSING = "session_missing"       # artifact says active, session dead
  SESSION_ORPHANED = "session_orphaned"     # session alive, no artifact reference
  STATUS_MISMATCH = "status_mismatch"       # artifact status != session reality
  STALE_REVIEW = "stale_review"             # review-index stale vs current head
  WORKFLOW_STATE_CHANGED = "workflow_state_changed"  # state.yaml differs from launch-time snapshot

class DriftEntry(BaseModel):
  kind: DriftKind
  role: Role | None
  expected: str
  actual: str
  detail: str

class ReconciliationReport(BaseModel):
  artifact_id: str
  timestamp: datetime
  drift: list[DriftEntry]
  healthy: bool
```

`reconcile()` re-reads `state.yaml` at call time and compares against the `WorkflowContext` that was used to spawn sessions. This detects the most operationally important drift: an external actor (operator, spec-driver CLI) changing workflow state while a session is running — e.g. `spec-driver workflow block` issued while a reviewer is active.

The caller decides action — re-spawn a dead session, kill an orphan, abort a session whose workflow state was blocked. autobahn provides diagnosis, not prescription.

Session state is advisory; artifact state is authoritative.

## 13. Vertical Slices

### Slice 1: Session lifecycle (no spec-driver CLI dependency)

Proves the core architecture — artifact loading, transition planning, adapter composition, session management — without depending on the review CLI that doesn't exist yet.

**Scenario:** A delta is in `awaiting_handoff` state. autobahn loads context, computes transition, spawns a session, observes it, and reconciles.

```python
# 1. Preflight
errors = check_prerequisites(policy, harness=harness, backend=backend)
if errors:
  raise errors[0]

# 2. Load
ctx = load_context(artifact_dir)

# 3. Plan
plan = transition_from_handoff(ctx)

# 4. Spawn
result = await spawn_role_session(ctx, plan,
  policy=policy, harness=harness, backend=backend)
handle = result.value

# 5. Observe
async for metadata in observe_session(handle, backend=backend):
  if not metadata.alive:
    break

# 6. Reconcile
report = await reconcile(ctx, policy=policy, backend=backend)
```

**Test strategy:**

- **Unit tests:** Model validation against fixtures, artifact loading, transition planning for each state/role
- **Mock integration tests:** Supervisor spawn flow with mock harness + mock backend
- **Real backend test:** Spawn a trivial subprocess (e.g. `echo "done"`) via the subprocess backend, observe completion, verify session lifecycle. This proves the backend abstraction works against a real process, not just mocks.
- **Contract tests:** Parse spec-driver-exported fixture YAML with autobahn's pydantic models. These tests run in autobahn's CI and break if schemas drift.

### Slice 2: Review flow (gated on spec-driver review CLI — OQ-001)

Adds `prime_review` and `verify_review_outcome` on top of slice 1. Requires spec-driver to define and implement the review CLI contract.

**Deferred until:** spec-driver delivers the review CLI (OQ-001, OQ-003).

## 14. Design Decisions Summary

| ID | Decision | Rationale |
|----|----------|-----------|
| DEC-001 | Narrow pydantic models (stable field subset) validated by contract tests | Consume only fields autobahn needs, `extra="ignore"`. Contract tests catch drift at CI time. Converge when DE-107 delivers shared schemas. |
| DEC-002 | Sync API for artifact ops, async for session lifecycle | File reads are sync, process management is async. No unnecessary async tax. |
| DEC-003 | Sandbox is policy/config, not protocol | RE-001 revises brief §5.5. Nix flake, srt, and harness internals already handle it. |
| DEC-004 | Three-layer ownership | spec-driver configures → nix/srt sandboxes → autobahn orchestrates. |
| DEC-005 | Thin baseline harness + optional typed extensions | Baseline for all harnesses; pi-mono gets richer support via composable Protocol extensions. |
| DEC-006 | 4-op async session backend (v1) | Minimal: create, is_alive, terminate, get_metadata. Resume/pause/attach deferred. |
| DEC-007 | Caller-injected adapters, no registry | Keeps library stateless, composable, testable. |
| DEC-008 | Structured mutations via spec-driver CLI | Schema authority stays with spec-driver. autobahn does not author spec-driver's API. |
| DEC-009 | Review state machine owned by spec-driver | autobahn depends on the contract, does not define it. |
| DEC-010 | Structured references, not prompts | spec-driver owns prompt content. TransitionPlan carries artifact_id, phase_id, handoff_path. |
| DEC-011 | Single-writer sessions.yaml (provisional) | Proposed precondition. Full authority definition blocked on OQ-005 cross-repo contract. |
| DEC-012 | v1 API is spawn/observe/terminate only | Honest about what v1 proves. Resume/pause require protocol extensions. |
| DEC-013 | Observation is structured logging in v1 | No custom event subsystem until a consumer exists. OTel as future path. |
| DEC-014 | Review APIs internal/experimental | Not exported until spec-driver review contract lands. Prevents API churn. |
| DEC-015 | Narrow read surface, stable field subset only | `extra="ignore"`, fields enumerated per file. Limits coupling to what autobahn actually uses. |
