# DR-004 – Review operations

## 1. Executive Summary

- **Delta**: [DE-004](./DE-004.md)
- **Status**: draft
- **Last Updated**: 2026-03-23
- **Synopsis**: Implement autobahn's review operations API by importing spec-driver's domain logic and types directly via `spec_driver.workflow`. Shared types become aliases, not duplicates. Exceptions propagate, not map. Review enums retire from autobahn. Three new public API functions: `prime_review`, `summarize_review_outcome`, `disposition_finding`.

## 2. Problem & Constraints

- **Current Behaviour**: autobahn can spawn agent sessions and observe/terminate them (DE-001/DE-002 "slice 1"), but has no review operations. It cannot prime a review, inspect review outcome, or disposition findings. The reviewer role is unreachable.
- **Drivers / Inputs**:
  - DR-001 §8 sketched review APIs gated on OQ-001 (spec-driver review CLI contract)
  - OQ-001 answered by DE-108 (CLI JSON envelope) and DE-109 (review state machine)
  - ADR-002 redirects integration from CLI subprocess to Python library imports
  - spec-driver delta (TBD) will create `spec_driver.workflow` public API surface
- **Constraints / Guardrails**:
  - ADR-001: spec-driver is schema authority — autobahn conforms, does not reimplement
  - ADR-002: orchestrator uses Python imports, not CLI subprocess
  - No duplication: shared types are aliases or direct imports, never parallel definitions
  - No translation: spec-driver exceptions propagate to callers, not mapped to autobahn equivalents
  - spec-driver must export higher-level operations, not just IO primitives (hard requirement)
- **Out of Scope**:
  - Review session orchestration (spawn reviewer, observe, terminate based on outcome)
  - Automatic finding disposition (agent-driven waive/defer policy)
  - Round session metadata schema
  - Tool adapter / subprocess invocation (superseded by ADR-002)

## 3. Architecture Intent

### 3.1 Governing principles

1. **Don't duplicate, don't map — import or alias.** Types that spec-driver owns are imported directly. autobahn does not maintain parallel definitions. Where backward compatibility requires a symbol in an autobahn module, it's a re-export alias.

2. **Exceptions propagate.** spec-driver's `StateNotFoundError`, `ReviewTransitionError`, `FindingsNotFoundError`, etc. are part of the public API surface. autobahn callers catch them directly. autobahn's own error taxonomy (`PreconditionError`, `BackendTransportError`, etc.) is for autobahn-specific concerns (sessions, backends), not for wrapping spec-driver domain errors.

3. **spec-driver exports operations, not just primitives.** The spec-driver public API must include higher-level functions (e.g., `prime_review_index(delta_dir, repo_root)`) that encapsulate multi-step orchestration. autobahn must not reconstruct the 100-line prime sequence from raw IO functions — that would violate the responsibility boundary and create a parallel maintenance burden.

### 3.2 Responsibility boundary

```
spec_driver.workflow                      autobahn/api/functions.py
────────────────────                      ──────────────────────────
Domain operations:                        Orchestration wrappers:
  prime_review_index(delta_dir, root)       prime_review() → ReviewBootstrapState
  read_findings(delta_dir)                  summarize_review_outcome() → OperationResult[ReviewOutcomeSummary]
  update_finding_disposition(...)           disposition_finding() → OperationResult
  can_approve(blocking)
  collect_blocking_findings(rounds)       Lifecycle integration:
  apply_review_transition(...)              load_context(), spawn_role_session(), etc.

Types & exceptions:                       Types (autobahn-owned only):
  BootstrapStatus, ReviewStatus, ...        ReviewBootstrapState
  FindingDisposition, ReviewFinding         ReviewOutcomeSummary
  StateNotFoundError, ReviewTransitionError OperationResult[T]
  FindingsNotFoundError, etc.
```

### 3.3 Import graph

```
autobahn/api/functions.py
  → spec_driver.workflow        (operations, types, exceptions)
  → autobahn/models/runtime.py  (ReviewBootstrapState, ReviewOutcomeSummary)

autobahn/models/artifacts.py
  → spec_driver.workflow        (BootstrapStatus, ReviewStatus, FindingStatus, etc.)
  → autobahn/models/enums.py   (WorkflowStatus, Role, PhaseStatus, etc.)

autobahn/models/enums.py
  → spec_driver.workflow        (re-export aliases for review enums — backward compat only)
```

No circular dependencies. spec-driver is a leaf dependency.

### 4.4 `autobahn/api/functions.py` — new functions

#### `prime_review`

```python
def prime_review(
  delta_id: str,
  *,
  root: Path,
) -> ReviewBootstrapState:
  """Orchestrate review priming via spec-driver domain logic.

  Evaluates staleness, builds/refreshes review index, writes
  review-index.yaml + review-bootstrap.md, transitions judgment
  to in_progress.

  Raises:
    spec_driver.workflow.StateNotFoundError: No workflow state for delta.
    spec_driver.workflow.StateValidationError: Invalid workflow state.
    spec_driver.workflow.ReviewIndexValidationError: Index validation failed.
  """
```

**Implementation (DEC-004-006):** Calls spec-driver's higher-level prime operation directly. The spec-driver public API must export a function that encapsulates the full prime sequence (staleness evaluation → index build → file writes → judgment transition). autobahn wraps the result in `ReviewBootstrapState`.

If spec-driver exposes `prime_review_index(delta_dir, repo_root) → PrimeResult`, autobahn's implementation is approximately:

```python
def prime_review(delta_id: str, *, root: Path) -> ReviewBootstrapState:
  delta_dir = _resolve_delta_dir(delta_id, root)
  result = spec_driver.workflow.prime_review_index(delta_dir, root)
  return ReviewBootstrapState(
    delta_id=delta_id,
    action=result.action,
    bootstrap_status=result.bootstrap_status,
    judgment_status=result.judgment_status,
    review_round=result.review_round,
    index_path=result.index_path,
    bootstrap_path=result.bootstrap_path,
  )
```

#### `summarize_review_outcome`

```python
def summarize_review_outcome(
  delta_dir: Path,
) -> OperationResult[ReviewOutcomeSummary]:
  """Read review findings and return outcome digest.

  Returns failed result if review not yet started or no findings exist.

  Raises:
    spec_driver.workflow.FindingsVersionError: Incompatible findings schema.
  """
```

**Note (adversarial finding #2):** Takes `delta_dir: Path`, not `WorkflowContext`. This function delegates entirely to spec-driver's domain logic (`read_findings`, `collect_blocking_findings`, `can_approve`). Passing `WorkflowContext` would imply autobahn's loaded artifact models are used, creating a hybrid type system. The path-based signature is honest about what happens: spec-driver reads its own files with its own types.

Implementation:
1. `read_findings(delta_dir)` → dict (spec-driver)
2. `read_review_index(delta_dir)` → dict (spec-driver, for judgment_status)
3. `collect_blocking_findings(rounds)` → list[ReviewFinding] (spec-driver)
4. `can_approve(blocking)` → (bool, reasons) (spec-driver)
5. Count findings by category and disposition status
6. Return `OperationResult(success=True, value=ReviewOutcomeSummary(...))`
7. On `FindingsNotFoundError`: return `OperationResult(success=False, error="...")`

#### `disposition_finding`

```python
def disposition_finding(
  delta_id: str,
  finding_id: str,
  *,
  action: FindingDispositionAction,
  root: Path,
  authority: DispositionAuthority = DispositionAuthority.AGENT,
  rationale: str | None = None,
  backlog_ref: str | None = None,
  resolved_at: str | None = None,
  superseded_by: str | None = None,
) -> OperationResult[None]:
  """Disposition a review finding.

  Eagerly validates per-action constraints before delegating to
  spec-driver's update_finding_disposition.

  Raises:
    spec_driver.workflow.FindingsNotFoundError: No findings file.
    spec_driver.workflow.FindingsVersionError: Incompatible schema.
  """
```

**Eager validation (adversarial finding #5):** Validates constraints before writing:
- `waive`: requires `rationale`
- `defer`: requires `rationale` and `backlog_ref`
- `fix`: requires `resolved_at` for blocking findings
- `supersede`: requires `superseded_by`

Returns `OperationResult(success=False, error="rationale required for waive")` on constraint violation rather than writing an invalid disposition and discovering it at approval time.

### Test strategy

Tests verify **autobahn's orchestration behaviour**, not spec-driver's domain logic:
- `prime_review`: mock file system or use temp dirs with fixture data; verify `ReviewBootstrapState` construction
- `summarize_review_outcome`: fixture findings data → verify counting, `can_approve` delegation, `OperationResult` wrapping
- `disposition_finding`: verify eager constraint validation; verify delegation to `update_finding_disposition`; verify `OperationResult` on success/failure

Do **not** re-test: staleness evaluation, approval guards, disposition constraint enforcement, transition tables — those are spec-driver's responsibility.

## 6. Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| DEC-004-001 | Direct Python import via `spec_driver.workflow`, no CLI subprocess | ADR-002. No serialization round-trip. |
| DEC-004-002 | `prime_review(delta_id, *, root)` signature | Maps to what spec-driver needs. `WorkflowContext` not consumed — spec-driver does its own IO. `git_head` dropped — derived internally. |
| DEC-004-003 | `summarize_review_outcome(delta_dir)` — path, not context | Delegates entirely to spec-driver. Avoids hybrid type system (autobahn artifact models + spec-driver domain types). |
| DEC-004-004 | Single `disposition_finding()` with action parameter, eager validation | Mirrors spec-driver's orchestration. Validates constraints before writing — better orchestrator UX than discovering invalid disposition at approval time. |
| DEC-004-005 | Review enums aliased from `spec_driver.workflow`, not duplicated | ADR-002 consequence. `enums.py` re-exports for backward compat. New code imports from canonical source. |
| DEC-004-006 | spec-driver MUST export higher-level operations | Hard requirement. autobahn must not reconstruct the ~100-line prime sequence from primitives. The spec-driver delta must include `prime_review_index()` or equivalent. |
| DEC-004-007 | No tool adapter layer | `adapters/tool/` unnecessary under ADR-002. |
| DEC-004-008 | No exception mapping | spec-driver exceptions propagate. autobahn error taxonomy is for autobahn concerns (sessions, backends). |
| DEC-004-009 | Shared types are aliases, not duplicates | Import or re-export, never maintain parallel definitions. |
