# DR-004 â€“ Review operations

## 1. Executive Summary

- **Delta**: [DE-004](./DE-004.md)
- **Status**: draft
- **Last Updated**: 2026-03-23
- **Synopsis**: Implement autobahn's review operations API by importing spec-driver's domain logic directly via `spec_driver.workflow`, retiring duplicate review enums, and exposing `prime_review`, `summarize_review_outcome`, and `disposition_finding` as public API functions.

## 2. Problem & Constraints

- **Current Behaviour**: autobahn can spawn agent sessions and observe/terminate them (DE-001/DE-002 "slice 1"), but has no review operations. It cannot prime a review, inspect review outcome, or disposition findings. The reviewer role is unreachable.
- **Drivers / Inputs**:
  - DR-001 Â§8 sketched review APIs gated on OQ-001 (spec-driver review CLI contract)
  - OQ-001 answered by DE-108 (CLI JSON envelope) and DE-109 (review state machine)
  - ADR-002 redirects integration from CLI subprocess to Python library imports
  - spec-driver delta (TBD) will create `spec_driver.workflow` public API surface
- **Constraints / Guardrails**:
  - ADR-001: spec-driver is schema authority â€” autobahn conforms, does not reimplement
  - ADR-002: orchestrator uses Python imports, not CLI subprocess
  - Library-first: no new external dependencies
  - spec-driver owns domain operations; autobahn does not recompose workflow logic from primitives
- **Out of Scope**:
  - Review session orchestration (spawn reviewer, observe, terminate based on outcome)
  - Automatic finding disposition (agent-driven waive/defer policy)
  - Round session metadata schema
  - Tool adapter / subprocess invocation (superseded by ADR-002)

autobahn's review functions are thin wrappers that:
1. Call spec-driver domain operations
2. Map results into autobahn's own orchestration types (`ReviewBootstrapState`, `OperationResult`)
3. Map exceptions into autobahn's error taxonomy where appropriate

autobahn does **not**:
- Reimplement staleness evaluation, approval guards, or disposition logic
- Maintain parallel enum definitions for review-related types
- Parse JSON envelopes or interpret exit codes

### 4.4 `autobahn/api/functions.py` â€” new functions

#### `prime_review`

```python
def prime_review(
  delta_id: str,
  *,
  root: Path,
) -> ReviewBootstrapState:
  """Orchestrate review priming via spec-driver domain logic.

  Evaluates staleness, builds/refreshes review index, writes
  review-index.yaml + review-bootstrap.md, transitions judgment
  to in_progress.

  Raises:
    PreconditionError: No workflow state, invalid state.
    ArtifactParseError: Malformed workflow files.
  """
```

Implementation delegates to spec-driver functions:
1. `read_state(delta_dir)` â€” get current workflow state
2. `get_head_sha(root)` â€” current git HEAD
3. `read_review_index(delta_dir)` (if exists) â€” check cache
4. `evaluate_staleness(...)` â€” derive cache status
5. `build_review_index(...)` â€” construct index data
6. `write_review_index(delta_dir, index_data)` â€” atomic write
7. `apply_review_transition(NOT_STARTED, BEGIN_REVIEW)` â€” judgment â†’ in_progress
8. Return `ReviewBootstrapState` with results

This mirrors what the CLI's `review_prime_command` does, minus the presentation layer. The spec-driver delta should ideally hoist this orchestration into a callable function (e.g., `spec_driver.workflow.prime_review_index(...)`) so autobahn doesn't reconstruct it from primitives. If not, autobahn composes the sequence from the public API surface â€” acceptable for v1 but creates a maintenance burden.

**Design decision (DEC-004-006):** Prefer spec-driver exporting a higher-level operation. If that's not available at implementation time, autobahn may compose from public primitives with a `# TODO: replace with spec-driver operation when available` marker.

#### `summarize_review_outcome`

```python
def summarize_review_outcome(
  context: WorkflowContext,
) -> OperationResult[ReviewOutcomeSummary]:
  """Read review findings and return outcome digest.

  Returns failed result if review not yet started or no findings exist.
  """
```

Implementation:
1. Derive `delta_dir` from `context.artifact_dir`
2. Call `read_findings(delta_dir)` from spec-driver
3. Call `collect_blocking_findings(rounds)` from spec-driver
4. Call `can_approve(blocking)` from spec-driver
5. Read judgment status from review-index (if available)
6. Construct `ReviewOutcomeSummary`

#### `disposition_finding`

```python
def disposition_finding(
  delta_id: str,
  finding_id: str,
  *,
  action: FindingDispositionAction,
  root: Path,
  authority: DispositionAuthority = DispositionAuthority.AGENT,
  rationale: str | None = None,
  backlog_ref: str | None = None,
  resolved_at: str | None = None,
  superseded_by: str | None = None,
) -> OperationResult[None]:
  """Disposition a review finding.

  Validates constraints (rationale required for waive/defer,
  backlog_ref required for defer, etc.) and delegates to
  spec-driver's update_finding_disposition.
  """
```

**Design decision (DEC-004-004):** Single function with `action` parameter rather than 4 separate functions (`resolve_finding`, `defer_finding`, etc.). The per-action parameter requirements (rationale, backlog_ref, resolved_at, superseded_by) are validated by the function. This matches how spec-driver's `_disposition_finding` works â€” a single orchestration function with a disposition dict.

### Test approach for review API functions

The review functions delegate to spec-driver domain logic. Tests should:
- Mock file system (not spec-driver functions) to test autobahn's orchestration wrapper
- Use spec-driver's own fixture data where available
- Verify autobahn-specific behaviour: error mapping, `OperationResult` wrapping, type construction

Tests should **not** re-test spec-driver's domain logic (staleness evaluation, approval guards, disposition constraints) â€” that's spec-driver's test suite responsibility.

## 6. Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| DEC-004-001 | Direct Python import via `spec_driver.workflow`, no CLI subprocess | ADR-002. Eliminates serialization round-trip, exit code mapping, JSON envelope parsing. |
| DEC-004-002 | `prime_review(delta_id, *, root)` signature | CLI-like inputs (delta ID + repo root). `WorkflowContext` not needed â€” spec-driver does its own state loading. `git_head` dropped â€” spec-driver derives HEAD internally. |
| DEC-004-003 | `summarize_review_outcome(context)` reads loaded context | Pure query, no mutation. Delegates to spec-driver's `read_findings`, `collect_blocking_findings`, `can_approve` for domain logic. |
| DEC-004-004 | Single `disposition_finding()` with action parameter | Mirrors spec-driver's single orchestration function. Per-action parameter validation inside. Avoids 4 near-identical functions. |
| DEC-004-005 | Review enums retired from autobahn, imported from `spec_driver.workflow` | ADR-002 consequence. No point maintaining parallel copies when directly importing. Non-review enums remain as transcriptions (used in YAML artifact models). |
| DEC-004-006 | spec-driver owns domain operations; autobahn wraps in `OperationResult` | Clear responsibility boundary. autobahn does not reimplement staleness, guards, or disposition logic. |
| DEC-004-007 | No tool adapter layer created | `adapters/tool/` from original DE-004 scope is unnecessary under ADR-002. `ToolInvocationError`/`ToolContractError` remain in error taxonomy for future external tool integration but are unused by review operations. |
