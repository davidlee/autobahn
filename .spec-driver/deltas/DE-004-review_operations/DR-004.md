# DR-004 – Review operations

## 1. Executive Summary

- **Delta**: [DE-004](./DE-004.md)
- **Status**: draft
- **Last Updated**: 2026-03-23
- **Synopsis**: Implement autobahn's review operations as thin wrappers around spec-driver's composed domain operations (`spec_driver.workflow`). Shared types imported directly — no duplicates, no translation. Exceptions propagate — no mapping. Review enums retired from autobahn. Dead tool-adapter error types removed.

## 2. Problem & Constraints

- **Current Behaviour**: autobahn can spawn agent sessions and observe/terminate them (DE-001/DE-002 "slice 1"), but has no review operations. The reviewer role is unreachable.
- **Drivers / Inputs**:
  - DR-001 §8 sketched review APIs gated on OQ-001
  - OQ-001 answered by DE-108/DE-109
  - ADR-002 redirects integration to Python library imports
  - DR-124/DE-124 creates `spec_driver.workflow` with composed operations, types, exceptions
- **Constraints / Guardrails**:
  - ADR-001: spec-driver is schema authority
  - ADR-002: orchestrator uses Python imports, not CLI subprocess
  - Don't duplicate, don't map — import or alias
  - Exceptions propagate — callers catch spec-driver exceptions directly
  - spec-driver exports operations, not just primitives (DE-124 hard requirement)
  - Disposition validation is spec-driver domain logic (DR-109), not autobahn policy
- **Out of Scope**:
  - Review session orchestration (spawn, observe, terminate)
  - Automatic finding disposition policy
  - Round session metadata schema
  - Tool adapter / subprocess invocation

## 3. Architecture Intent

### 3.1 Governing principles

1. **Don't duplicate, don't map — import or alias.** Types that spec-driver owns are imported directly. No parallel definitions. Re-export aliases for backward compatibility only.

2. **Exceptions propagate.** spec-driver's `StateNotFoundError`, `ReviewTransitionError`, `ReviewApprovalGuardError`, `FindingNotFoundError`, etc. are the error contract. autobahn callers catch them. autobahn's own error taxonomy is for autobahn-specific concerns (sessions, backends).

3. **Use spec-driver result types directly.** `PrimeResult`, `ReviewSummary`, `DispositionResult` from `spec_driver.workflow` — no autobahn wrapper types. If autobahn later needs orchestration-specific annotations, it can wrap then.

4. **Disposition validation is domain logic.** Constraints like "waive requires rationale" and "defer requires backlog_ref" are review state machine rules (DR-109). They're enforced in spec-driver's `disposition_finding` operation (DE-124), not reimplemented in autobahn.

### 3.2 Responsibility boundary

```
spec_driver.workflow                      autobahn/api/functions.py
────────────────────                      ──────────────────────────
Composed operations:                      Thin wrappers:
  prime_review(delta_dir, repo_root)        prime_review(delta_dir, repo_root)
    → PrimeResult                             call spec-driver, return directly
  summarize_review(delta_dir)               summarize_review_outcome(delta_dir)
    → ReviewSummary                           call spec-driver, return directly
  disposition_finding(delta_dir, ...)       disposition_finding(delta_dir, ...)
    → DispositionResult                       call spec-driver, return directly
  complete_review(delta_dir, ...)           (not wrapped — see DEC-004-013)
  teardown_review(delta_dir)              Lifecycle integration (existing):
                                            load_context(), spawn_role_session(), etc.
Types & exceptions:
  BootstrapStatus, ReviewStatus, ...
  PrimeResult, ReviewSummary, ...
  StateNotFoundError, ReviewApprovalGuardError, ...
```

autobahn's wrappers call the spec-driver operation and return its result. All take `delta_dir: Path` — the caller resolves delta ID to path upstream (e.g., from `WorkflowContext.artifact_dir` after `load_context()`). No result type wrapping. No exception translation. No validation reimplementation.

**`complete_review` is deliberately not wrapped (DEC-004-013).** Review completion is a reviewer-agent action — the agent calls the CLI (`spec-driver review complete`) as part of its session, writing its own summary. The orchestrator observes the outcome via `summarize_review_outcome` after the session ends. If a future use case requires the orchestrator to complete reviews programmatically, a wrapper can be added then.

## 4. Code Impact Summary

### 4.1 `autobahn/models/enums.py` — retire review enums

```python
  FindingDispositionAction,
  DispositionAuthority,

```

Retained as autobahn-owned transcriptions (unchanged):
- `Role`, `WorkflowStatus`, `PhaseStatus`, `ArtifactKind`
- `SessionStatus`, `NextActivityKind`, `HandoffTransitionStatus`

### 4.2 `autobahn/models/artifacts.py` — update imports

Review-related models (`ReviewBlock`, `FindingDisposition`, `Finding`, `ReviewRound`, etc.) import types from `spec_driver.workflow`. Since `enums.py` re-exports them, existing internal imports continue to work.

### 4.3 `autobahn/models/errors.py` — remove dead types

Remove `ToolInvocationError` and `ToolContractError`. No code uses them (DEC-004-012).

### 4.4 `autobahn/models/runtime.py` — no new types

`ReviewBootstrapState` and `ReviewOutcomeSummary` from the original DR-004 are not created. autobahn uses spec-driver's `PrimeResult` and `ReviewSummary` directly (DEC-004-010).

### 4.5 `autobahn/api/functions.py` — new functions

#### `prime_review`

```python
from spec_driver.workflow import (
  prime_review as _sd_prime_review,
  PrimeResult,
)

def prime_review(
  delta_dir: Path,
  repo_root: Path,
) -> PrimeResult:
  """Prime a review via spec-driver.

  Evaluates staleness, builds/refreshes review index, writes
  review-index.yaml + review-bootstrap.md, transitions judgment
  to in_progress.

  Raises:
    spec_driver.workflow.StateNotFoundError
    spec_driver.workflow.StateValidationError
    spec_driver.workflow.ReviewIndexValidationError
  """
  return _sd_prime_review(delta_dir, repo_root)
```

#### `summarize_review_outcome`

```python
from spec_driver.workflow import (
  summarize_review as _sd_summarize,
  ReviewSummary,
)

def summarize_review_outcome(
  delta_dir: Path,
) -> ReviewSummary:
  """Query review findings and return outcome digest.

  Raises:
    spec_driver.workflow.FindingsNotFoundError
    spec_driver.workflow.FindingsVersionError
  """
  return _sd_summarize(delta_dir)
```

#### `disposition_finding`

```python
from spec_driver.workflow import (
  disposition_finding as _sd_disposition,
  FindingDispositionAction,
  DispositionAuthority,
  DispositionResult,
)

def disposition_finding(
  delta_dir: Path,
  finding_id: str,
  *,
  action: FindingDispositionAction,
  authority: DispositionAuthority = DispositionAuthority.AGENT,
  rationale: str | None = None,
  backlog_ref: str | None = None,
  resolved_at: str | None = None,
  superseded_by: str | None = None,
) -> DispositionResult:
  """Disposition a review finding.

  Constraint validation (rationale for waive, backlog_ref for defer,
  etc.) is enforced by spec-driver's operation — not reimplemented here.
  Note: backlog_ref is only required for blocking finding deferrals
  per DR-109 §3.4 — non-blocking deferrals need only rationale.

  Raises:
    spec_driver.workflow.FindingsNotFoundError
    spec_driver.workflow.FindingsVersionError
    spec_driver.workflow.FindingNotFoundError
  """
  return _sd_disposition(
    delta_dir,
    finding_id,
    action=action,
    authority=authority,
    rationale=rationale,
    backlog_ref=backlog_ref,
    resolved_at=resolved_at,
    superseded_by=superseded_by,
  )
```

All functions take `delta_dir: Path` consistently (DEC-004-002). The caller resolves delta ID → path upstream, typically via `WorkflowContext.artifact_dir` from `load_context()`. Parameter name `repo_root` matches spec-driver's convention (DEC-004-014).

### Test strategy

autobahn's wrappers are thin — tests verify:
- `delta_id` → `delta_dir` resolution
- Correct delegation to spec-driver operation (mock the operation, verify call args)
- Return value passthrough (spec-driver result returned as-is)
- Exception propagation (spec-driver exceptions reach caller)

Tests do **not** re-test: staleness evaluation, approval guards, disposition constraints, finding status derivation — that's spec-driver's test suite.

## 6. Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| DEC-004-001 | Direct Python import via `spec_driver.workflow` | ADR-002. No serialization round-trip. |
| DEC-004-002 | All wrappers take `delta_dir: Path` consistently | Caller resolves ID upstream (from `WorkflowContext.artifact_dir`). No redundant resolution in wrappers. |
| DEC-004-003 | `summarize_review_outcome(delta_dir)` — path, not context | Delegates entirely to spec-driver. No hybrid type system. |
| DEC-004-004 | `disposition_finding()` with typed kwargs | Mirrors spec-driver's typed signature. No `dict` passthrough. |
| DEC-004-005 | Review enums aliased from `spec_driver.workflow` | No parallel copies. Re-exports for backward compat. |
| DEC-004-006 | spec-driver MUST export higher-level operations | Hard requirement. Resolved by DE-124. |
| DEC-004-007 | No tool adapter layer | Superseded by ADR-002. |
| DEC-004-008 | No exception mapping | spec-driver exceptions propagate to callers. |
| DEC-004-009 | Shared types are aliases, not duplicates | Import or re-export, never parallel definitions. |
| DEC-004-010 | Use spec-driver result types directly | `PrimeResult`, `ReviewSummary`, `DispositionResult` — no autobahn wrappers. |
| DEC-004-011 | Disposition validation is spec-driver domain logic | Constraints enforced in DE-124. Note: `backlog_ref` only required for blocking finding deferrals per DR-109 §3.4. |
| DEC-004-012 | Remove dead `ToolInvocationError`/`ToolContractError` | Designed for CLI subprocess approach that ADR-002 supersedes. No code uses them. |
| DEC-004-013 | `complete_review` deliberately not wrapped | Review completion is a reviewer-agent action via CLI. Orchestrator observes outcome via `summarize_review_outcome`. |
| DEC-004-014 | Standardise on `repo_root` parameter name | Matches spec-driver convention. |

### Development dependency model

autobahn and spec-driver share a uv workspace. `pyproject.toml` already declares:

```toml
[tool.uv.workspace]
members = ["spec-driver"]

[tool.uv.sources]
spec-driver = { workspace = true, editable = true }
```

`uv sync` installs both packages in the same virtualenv. No additional CI or dependency configuration is needed — this is the existing setup from DE-001.
