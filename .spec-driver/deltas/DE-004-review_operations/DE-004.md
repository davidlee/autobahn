## 1. Summary & Context

- **Predecessor**: DE-003 (schema reconciliation), DE-002 (orchestration patterns)
- **Implementation Plan**: [IP-004](./IP-004.md)
- **Design Revision**: [DR-004](./DR-004.md)
- **Change Drivers**: spec-driver DE-108/DE-109 provide the review CLI contract that DR-001 §8 gated review operations on (OQ-001)

## 2. Motivation

- DR-001 defined `prime_review` and `verify_review_outcome` as internal/experimental APIs, gated on spec-driver's review CLI contract (OQ-001)
- OQ-001 is now answered: DE-108 provides `--format json` with versioned envelope; DE-109 provides the review state machine, judgment lifecycle, and finding disposition commands
- autobahn's "second vertical slice" (review flow) can now be implemented
- Without review operations, autobahn cannot orchestrate the reviewer role — it can spawn a reviewer session but cannot bootstrap review context or record outcomes

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Tool invocation adapter for spec-driver CLI (JSON envelope parsing, exit code handling)
  - `prime_review(context, *, git_head)` — invoke `spec-driver review prime --format json`, return bootstrap state
  - `verify_review_outcome(context)` — read review-findings, validate consistency, return outcome summary
  - Finding disposition wrappers — resolve, defer, waive, supersede (invoke CLI, parse response)
  - `ReviewBootstrapState` and `ReviewOutcomeSummary` model types (previously unspecified per DR-001)
  - Public API exports for review operations
- **Operational Constraints**: Library-first, no new dependencies. Tool invocation via subprocess (same pattern as existing harness adapters).
- **Dependencies**: DE-003 (schema reconciliation) must complete first — models must be correct

## 4. Out of Scope

- Review session orchestration (spawn reviewer, observe, terminate based on review outcome) — follow-up delta
- Automatic finding disposition (agent-driven waive/defer decisions) — policy question, not this delta
- Review round session metadata schema (autobahn owns per DEC-109-009; define shape when needed)
- Additional harness adapters (pi-mono, Codex)

## 5. Approach Overview

- **Key Changes**:
  - `adapters/tool/protocol.py` (new) — tool invocation protocol (invoke CLI, parse JSON envelope)
  - `adapters/tool/spec_driver.py` (new) — spec-driver CLI adapter implementing the protocol
  - `models/runtime.py` — add `ReviewBootstrapState`, `ReviewOutcomeSummary`, `ToolInvocationError`, `ToolContractError`
  - `api/functions.py` — add `prime_review`, `verify_review_outcome`, finding disposition functions
  - `api/__init__.py` — export new public API functions

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `prime_review` invokes spec-driver CLI and returns typed `ReviewBootstrapState`
  - `verify_review_outcome` reads review-findings and returns `ReviewOutcomeSummary`
  - Finding disposition functions invoke CLI and return structured result
  - Error handling: precondition failures (exit 2), guard violations (exit 3), unexpected errors (exit 1)
  - `just check` passes

## 7. Risks & Mitigations

- **Risk**: spec-driver CLI not available in test environment — _Likelihood_: high — _Impact_: medium — _Mitigation_: Mock the tool adapter in tests; integration test optional
- **Risk**: JSON envelope format changes — _Likelihood_: low — _Impact_: medium — _Mitigation_: Version field in envelope; validate version before parsing

## 8. Follow-ups & Tracking

- Review session orchestration (spawn reviewer with primed context, observe, handle outcome)
- Round session metadata schema definition
- Additional harness adapters
