# AUD-001 — DE-001 Conformance Audit

## Scope

Post-implementation conformance audit for DE-001 (Repository bootstrap and
architecture foundation). Covers all 5 phases of IP-001, all code under
`autobahn/`, all tests under `tests/`, and alignment with DR-001.

## Findings

### F-001: `observe_session` uses `Supervisor.__new__` bypass [severity: medium]

**Location:** `autobahn/api/functions.py:108`

`observe_session` creates a half-initialized Supervisor via `__new__` to avoid
requiring a harness argument. This is a code smell — it breaks the Supervisor's
constructor contract and couples the API layer to Supervisor internals. If
Supervisor's `__init__` ever adds state beyond `harness` and `backend`, this
silently breaks.

**outcome:** defect
**disposition:** follow_up_backlog
**rationale:** Not blocking for v1 vertical slice, but should be resolved before
adding session lifecycle complexity. The clean fix is either: (a) extract
`observe` as a free function that takes only a backend, since it doesn't need
the harness, or (b) make Supervisor accept optional harness.

---

### F-002: `SubprocessBackend.create` hardcodes `Role.OTHER` and empty `artifact_id` [severity: low]

**Location:** `autobahn/adapters/session/subprocess_backend.py:55-56`

The backend creates SessionHandles with `role=Role.OTHER` and `artifact_id=""`
because it doesn't have access to the orchestration context. The caller
(Supervisor) doesn't fix these fields after creation.

**outcome:** defect
**disposition:** follow_up_backlog
**rationale:** The SessionHandle returned to the caller carries incorrect
metadata. Supervisor.spawn should populate role/artifact_id from context/plan
after receiving the handle from the backend. Not blocking since these fields
aren't consumed downstream in v1, but will cause confusion when sessions.yaml
writing is implemented.

---

### F-003: `SubprocessBackend` env handling drops parent environment [severity: medium]

**Location:** `autobahn/adapters/session/subprocess_backend.py:39`

When `launch.env` is non-empty, the code passes `env={**launch.env}` which
**replaces the entire environment** rather than augmenting it. The subprocess
won't inherit PATH, HOME, or any other ambient variables. When `launch.env` is
empty/None, `env=None` is passed, which correctly inherits. This asymmetry
means adding a single env var (like CLAUDE_TIMEOUT) breaks the subprocess.

**outcome:** defect
**disposition:** spec_patch
**rationale:** This is a real bug that will bite in production. The fix is
`env = {**os.environ, **launch.env} if launch.env else None`. Fixing now.

---

### F-004: `transition_from_handoff` hardcodes handoff path [severity: low]

**Location:** `autobahn/runtime/transition.py:39`

`handoff_path="workflow/handoff.current.yaml"` is hardcoded. If the caller uses
a different `workflow_dir` name (which `load_context` supports via its
`workflow_dir` parameter), the TransitionPlan will carry a wrong path.

**outcome:** defect
**disposition:** follow_up_backlog
**rationale:** Minor in v1 since the TransitionPlan's handoff_path is
informational (harnesses don't currently consume it for file I/O). Should be
derived from context when the path becomes load-bearing.

---

### F-005: `_TERMINAL_STATES` duplicated between transition.py and reconcile.py [severity: low]

**Location:** `autobahn/runtime/transition.py:12`, `autobahn/runtime/reconcile.py:22`

Both modules define their own terminal-states set (`{WorkflowStatus.APPROVED}`).
If the definition of "terminal" changes, two sites need updating.

**outcome:** observation
**disposition:** follow_up_backlog
**rationale:** Extract to a shared constant in models/enums.py or a
runtime/_constants.py. Low priority — single value today.

---

### F-006: `reconcile` accepts unused `policy` parameter [severity: low]

**Location:** `autobahn/runtime/reconcile.py:44`

The `policy: RuntimePolicy` parameter is accepted but never used in the
function body. It was included for API symmetry with other functions but
currently serves no purpose.

**outcome:** observation
**disposition:** aligned
**rationale:** Acceptable API reservation — reconciliation may need policy
context in the future (e.g., timeout for backend probes). Leaving it avoids
a breaking API change later. No action needed.

---

### F-007: No schema/version validation in artifact loader [severity: medium]

**Location:** `autobahn/artifacts/loader.py`

DR-001 §9 states: "autobahn validates against version 1 of each schema. Unknown
schema or unsupported version → ArtifactContractError, never silently degrade."
The loader does not check `schema` or `version` fields in the YAML data. It
relies solely on pydantic field validation, which will succeed even if the schema
marker is wrong or missing (since those fields aren't in the models, and
`extra="ignore"` drops them).

**outcome:** gap
**disposition:** follow_up_backlog
**rationale:** DR-001 is explicit about this requirement. The fix is
straightforward (validate schema marker and version before model_validate). Not
blocking the vertical slice since all fixtures use v1 schemas, but this is a
real contract gap that should be addressed before consuming non-fixture data.

---

### F-008: `reconcile` function in reconcile.py imports from adapters/ [severity: low]

**Location:** `autobahn/runtime/reconcile.py:12`

DR-001 §5 coupling rules say "runtime/ depends on models/, adapters/ (via
protocols)". This is technically satisfied — reconcile.py imports
`SessionBackend` from `adapters/session/protocol.py`, which is a protocol
definition. However, `ReconciliationReport` and `DriftItem` are defined in
runtime/reconcile.py rather than models/. This is fine per the coupling rules
(runtime can define its own types), but these types will eventually need to
be importable by callers without pulling in runtime internals.

**outcome:** observation
**disposition:** aligned
**rationale:** Acceptable for v1. If external consumers need these types, they
can be moved to models/ or re-exported from api/.

---

### F-009: DR-001 specifies `reconcile` returns `OperationResult[ReconciliationReport]` but reconcile never returns a failed result [severity: low]

**Location:** `autobahn/runtime/reconcile.py:41-122`

The function always returns `OperationResult(success=True, ...)`. There is no
code path that returns `success=False`. The error semantics described in DR-001
(artifact re-read failure) would raise exceptions rather than returning a failed
result.

**outcome:** observation
**disposition:** aligned
**rationale:** The current behaviour is defensible — reconciliation I/O failures
are truly exceptional (broken filesystem) and should propagate as exceptions.
The OperationResult wrapper still provides a consistent API surface for callers
and leaves room for future "soft failure" paths.

---

### F-010: `ClaudeCodeAdapter.launch_spec` builds a prompt string rather than structured args [severity: low]

**Location:** `autobahn/adapters/harness/claude_code.py:45-47`

The prompt `"resume {plan.artifact_id} phase {plan.phase_id}"` is a free-form
string. Claude Code's actual CLI semantics for resuming work may differ. This is
a placeholder since the exact CLI contract isn't defined yet.

**outcome:** observation
**disposition:** aligned
**rationale:** Explicitly acknowledged in DR-001 §7 — the prompt/bootstrap seam
is harness-specific and will be refined when Claude Code's continuation semantics
are better understood.

---

### F-011: No sessions.yaml write support [severity: medium]

**Location:** N/A

DR-001 §9 specifies autobahn owns sessions.yaml writes. DE-001 §3 lists
"sessions.yaml" as autobahn-owned. The implementation reads sessions.yaml but
never writes to it. spawn_role_session doesn't persist session handles.

**outcome:** gap
**disposition:** follow_up_backlog
**rationale:** Explicitly deferred — OQ-005 (sessions.yaml ownership authority)
is still open. The vertical slice proves the architecture without persistence.
Sessions.yaml writes should be a follow-up delta once the cross-repo contract
with spec-driver is defined.

## Summary Assessment

### Architecture

The architecture is sound for a v1 vertical slice. The three-layer split
(models → artifacts/adapters → runtime → api) enforces clean coupling and the
Protocol-based adapter design is appropriately thin. Key strengths:

- **Coupling discipline is real.** The import coupling test enforces it
  mechanically, not just by convention. adapters/ genuinely depends only on
  models/.
- **Artifact-first recovery works.** The loader reads whatever exists on disk
  and constructs a valid WorkflowContext with None optionals. No magic state.
- **Reconciliation is honest.** It re-reads from disk (DEC-017) and reports
  drift without trying to auto-repair. The drift taxonomy is useful.
- **Error taxonomy has a clean split** between exceptions (invariant violations)
  and OperationResult (expected outcomes).

### Concerns

- **`Supervisor.__new__` bypass (F-001)** is the most architecturally worrying
  item. It suggests the Supervisor abstraction may be slightly wrong — observe
  genuinely doesn't need a harness, so bundling it into Supervisor is a leaky
  coupling. Easy fix but worth doing before the class grows.
- **Environment replacement bug (F-003)** is a real production defect that needs
  fixing before any real harness invocation.
- **No schema/version validation (F-007)** is a stated DR requirement that was
  skipped. It's the kind of gap that compounds — once you're consuming
  real-world files that may be v2 or from a different tool, silent parsing is
  dangerous.
- **No sessions.yaml write (F-011)** is the biggest functional gap, but it's
  correctly deferred behind OQ-005.

### Test Coverage

95 tests across 10 test files. Coverage is appropriate for a v1 slice:
- Contract tests anchor enum values and model parsing to spec-driver source
- Loader tests cover happy path plus all error paths
- API tests use mock adapters (quick, deterministic)
- Integration test proves real subprocess lifecycle
- Coupling test is a good structural guard

### Verdict

**Ready for closure** after F-003 is fixed (environment bug). All other
findings are correctly dispositioned as follow-up or aligned observations.
The architecture proves the concept cleanly and the test suite is solid for
the scope.
