  Second Design Question

  Async stance for the public API.

  Session lifecycle operations (spawn, observe, resume, terminate) are

  What is the session backend abstraction?

  subprocess)
  - Harness adapter вҖ” the agent tool (Claude Code, Codex, Gemini CLI)
  - Sandbox adapter вҖ” the isolation wrapper (bubblewrap, none)

  A. Layered composition вҖ” session wraps sandbox wraps harness

  - Clean nesting: session manages the window, sandbox manages isolation,
  harness manages the agent process

  - Con: Three protocol interfaces that must agree on how to pass through

  coordinates them directly
  - Session backend creates the session, sandbox wraps the command, harness
  provides the command
  - Pro: Simpler protocols вҖ” each adapter answers its own questions without
  knowing about the others
  - Pro: Easier to test вҖ” mock each independently
  - Con: Composition logic lives in the runtime layer rather than being enforced
   by types

  - Harness adapter knows how to produce a "sandboxed command" (delegates to
  sandbox internally)
  - Session backend just runs whatever command the harness gives it
  - Pro: Session backend stays very thin
  - Con: Couples harness and sandbox вҖ” harder to mix-and-match

  Recommendation: B. The runtime layer is the natural place for composition.
  Each adapter stays narrow вҖ” harness says "here's my launch command and env
  requirements," sandbox says "here's how to wrap a command," session backend
  says "here's how to run and track a process." The runtime layer stitches them.

в—Ҹ DR-001 Section 2: Architecture Intent

  Target Outcomes:
  - A Python library (autobahn) providing typed, coarse-grained orchestration of
   agent sessions driven by spec-driver workflow artifacts
  - Clean three-layer ownership model: spec-driver configures вҶ’ nix/srt
  sandboxes вҶ’ autobahn orchestrates
  - Pluggable harness adapters with tiered capability (baseline protocol +
  optional typed extensions)
  - Recoverable from any failure using only artifact state + adapter policy вҖ” no
   session durability assumptions

  Guiding Principles:

  1. Artifact-first continuity. Workflow artifacts are the only durable truth.
  Runtime state is derived, not primary. If autobahn crashes, the next
  invocation reads artifacts and reconstructs what to do.
  2. Thin protocols, rich adapters. Core protocols are minimal (4 ops for
  session backend, 2-3 for harness baseline). Richer harness capabilities
  (pi-mono's context engineering, sub-agents) compose via optional Protocol
  extensions вҖ” never bolted onto the base.
  3. autobahn doesn't configure agents. spec-driver installs skills, hooks,
  preboot, CLAUDE.md. Autobahn's job starts when the agent binary is ready to
  run. It provides the runtime prompt ("here's what to do now") not the static 
  bootstrap ("here's who you are").
  4. autobahn doesn't sandbox agents. The caller provides an executable вҖ”
  claude, jailed-claude, srt claude, whatever. Autobahn runs it. Sandbox policy
  metadata can travel in RuntimePolicy for logging/observation, but autobahn
  doesn't enforce it.
  5. Sync reads, async runs. Artifact consumption is synchronous pure
  computation. Session lifecycle is async. The boundary is natural and follows
  the data.

  State Transitions / Lifecycle:

  autobahn consumes but does not own the spec-driver workflow state machine (7
  states, 7 commands). It reads state.yaml to determine current state, reads
  handoff.current.yaml to compute what to do next, and writes back to
  sessions.yaml (runtime-owned) and potentially review-findings.yaml /
  review-index.yaml (where explicitly permitted).

  spec-driver state machine (read-only to autobahn):

    planned вҶ’ implementing вҶ’ awaiting_handoff вҶ’ reviewing
                  вҶ‘                                 в”Ӯ

  The tension is real: separate repos buy you boundary discipline and bounded

  Four options:

  A. Accept duplication, manual sync

  - autobahn owns copies of enums and YAML models
  - When spec-driver changes schemas, autobahn must be updated manually
  - Pro: Zero coupling, simplest repo topology
  - Con: Drift is inevitable. The "standard" I proposed ("enum values must
  match") is a policy enforced by humans вҖ” it will fail eventually

  B. Common schema package (third repo)

  - spec-driver-schemas or similar вҖ” thin package with just enums, pydantic
  models for the 5 YAML files, schema version constants
  - Both spec-driver and autobahn depend on it
  - Pro: Single source of truth for the contract
  - Con: Third repo to maintain, version, release. Coordination overhead for
  what might be 200 lines of models. May be premature.

  C. spec-driver publishes a thin stable extras package

  - spec-driver already exists as a package. Add an extras or sub-package like
  spec-driver[schemas] or supekku.schemas вҖ” a deliberately stable, minimal
  surface
  - autobahn depends on supekku.schemas only
  - Pro: No third repo. Schema authority stays with spec-driver. autobahn gets
  typed imports.
  - Con: Couples autobahn to spec-driver as a Python dependency (but only the
  stable schema surface, not internals). Requires spec-driver to explicitly
  maintain that stability boundary.

   вҖ” typed models in spec-driver that are designed for external consumption.

  D. Code generation from spec-driver's schema definitions

  - spec-driver's workflow_metadata.py already has structured field definitions
  (BlockMetadata). Generate pydantic models from those definitions, either into
  spec-driver itself or into autobahn.
  - Pro: Schema definitions stay in spec-driver, models are derived
  - Con: Codegen machinery to build and maintain. Feels over-engineered for the
  current scale.

  My recommendation: C, with timing awareness.

  The practical path:
  1. Now (v1): autobahn defines its own models. They're thin and close to the
  YAML structure. This is the "accept duplication short-term" pragmatic start.
  2. When DE-107 lands: spec-driver has pydantic models. We evaluate whether

  Design notes on the API shape

  Adapters are caller-provided, not global. The caller passes harness and
  backend explicitly. No service locator, no registry, no global state. This
  keeps the library composable and testable вҖ” tests pass mock adapters,
  production code passes real ones.

  Reconcile reports but doesn't repair. The brief says "inspect and repair
  drift" but auto-repair is dangerous in v1. A reconciliation report lets the
  caller (or future supervisor) decide what to do. Repair operations can be
  added as separate functions that accept a ReconciliationReport.

  observe_session is an async iterator. Natural for "watch this until it's done"
   without forcing the caller into a callback model. The caller can async for 
  metadata in observe_session(...) or break early.

  OperationResult wraps fallible operations. Session and reconciliation
  operations can fail in expected ways (session already dead, artifact dir
  missing). Result type keeps the API exception-light at the boundary while
  still typed.

  One thing I'm noting for the ADR list: the decision to make adapters
  caller-provided rather than registry-based is worth recording. It's a
  deliberate coupling choice.

  Doctrine candidate вҖ” ADR: "Adapters are injected by the caller, not resolved 
  from a registry." Keeps the library stateless and composable. Prevents hidden
  global configuration.

  Schema versioning

  All workflow files carry schema and version fields. autobahn validates against
   version 1 of each schema. If it encounters an unknown version, it raises
  rather than guessing.

  # artifacts/schema.py

  SUPPORTED_SCHEMAS = {
    "supekku.workflow.state": {1},
    "supekku.workflow.handoff": {1},
    "supekku.workflow.review-index": {1},
    "supekku.workflow.review-findings": {1},
    "supekku.workflow.sessions": {1},
  }

  def validate_schema_version(schema: str, version: int) -> None:
    supported = SUPPORTED_SCHEMAS.get(schema)
    if supported is None:
      raise UnknownSchemaError(schema)
    if version not in supported:
      raise UnsupportedSchemaVersionError(schema, version, supported)

  DR-001 Section 8: Vertical Slice вҖ” Reviewer Flow

  This is the end-to-end path that proves the architecture. One narrow flow
  through every layer.

  Scenario

  A delta is in awaiting_handoff state. The handoff targets the reviewer role.
  autobahn needs to: load the context, compute the transition, prime the
  reviewer, spawn a review session, and after the session completes, record the
  outcome.

  Flow

  Caller code (test or future CLI):

    # 1. Load
    ctx = load_context(artifact_dir)

    # 2. Plan
    plan = transition_from_handoff(ctx)
    # plan.target_role == Role.REVIEWER
    # plan.activity == NextActivityKind.REVIEW
    # plan.initial_prompt == "Review phase IP-001.PHASE-03 per ..."

    # 3. Prime
    git_head = get_current_head(ctx.artifact_dir)
    bootstrap = prime_review(ctx, git_head=git_head)
    # bootstrap.status in {BootstrapStatus.WARM, .STALE, .COLD}

    # 4. Spawn
    result = await spawn_role_session(
      ctx, plan,
      policy=RuntimePolicy(
        harness="claude_code",
        session_backend="tmux",
        work_dir=artifact_dir.parent,
      ),
      harness=ClaudeCodeAdapter(),
      backend=TmuxBackend(),
    )
    handle = result.value

    # 5. Observe
    async for metadata in observe_session(handle, backend=TmuxBackend()):
      # log progress, check for timeout
      if metadata.alive is False:
        break

    # 6. Record outcome
    outcome = harness.parse_exit(metadata.exit_code)
    record_review_outcome(
      ctx,
      status=ReviewStatus.CHANGES_REQUESTED,  # or APPROVED

    )

    # 7. Reconcile
    report = await reconcile(ctx, policy=policy, backend=TmuxBackend())

  How does autobahn learn the review outcome?

  - C. Agent writes a structured output file that autobahn reads вҖ” a lighter

  DR-001 Section 9: Sandbox Landscape (Position Statement)

  autobahn deliberately does not own sandbox orchestration. This section
  documents why and maps the landscape for future reference.

  Three sandbox layers exist in practice

  1. Harness-internal sandboxing
  - Claude Code uses bwrap (Linux) / sandbox-exec (macOS) internally
  - Codex uses its own sandbox-exec integration
  - The agent binary arrives pre-sandboxed from the harness vendor's perspective
  - autobahn cannot see or influence this layer

  2. Environment-level sandboxing (nix flake / jail-nix)
  - jailed-agents.nix composes agent + jail profile + env policy
  - Profiles control: network access, persistent home, git push blocking, git
  identity, postgres exposure, subagent depth
  - The jail wraps the entire agent binary вҖ” autobahn sees jailed-claude as an
  opaque executable
  - Policy-bearing: the nix flake is the sandbox authority on NixOS

  3. Standalone sandbox wrappers (Anthropic srt)
  - @anthropic-ai/sandbox-runtime вҖ” npm CLI/library
  - Wraps arbitrary commands with filesystem + network restrictions
  - Uses bwrap (Linux) / sandbox-exec (macOS) + proxy-based network filtering
  - TypeScript, not Python вҖ” usable as a command prefix (srt claude ...), not as
   a library autobahn would import
  - Research preview, API may evolve

  autobahn's position

  autobahn sits above all three layers. It receives an executable path from
  RuntimePolicy and runs it. Whether that executable is:

  - claude (unsandboxed)
  - jailed-claude (nix-jailed)
  - srt claude (srt-wrapped)
  - Something else entirely

  ...is the caller's decision. autobahn's RuntimePolicy.sandbox_profile field
  carries metadata (profile name, policy description) for logging and
  observation, but autobahn does not enforce, configure, or verify sandbox
  state.

  Why not a sandbox adapter protocol?

  - The nix flake already composes sandbox + agent at a higher level than
  autobahn can reach
  - Harness-internal sandboxing is opaque by design
  - srt is a CLI wrapper, not a composable Python protocol
  - Building a sandbox abstraction in autobahn would be a worse version of what
  already exists at each layer
  - The adapter protocol pattern is reserved for concerns autobahn genuinely
  orchestrates (harness launch, session management)

  Future considerations

  If a need arises for autobahn to verify sandbox state (e.g. "confirm the agent
   is running sandboxed before allowing write access to production"), that would
   be an observation concern вҖ” querying metadata about the running environment вҖ”
   not an orchestration concern. It could surface as an optional extension on
  the session backend or harness adapter.

  DR-001 Section 10: Observation & Reconciliation

  Observation

  Lightweight structured events emitted during orchestration. Not a logging
  framework вҖ” just typed records of what happened.

  # observation/events.py

  class EventKind(StrEnum):
    SESSION_SPAWNED = "session_spawned"
    SESSION_ENDED = "session_ended"
    SESSION_TIMEOUT = "session_timeout"
    REVIEW_PRIMED = "review_primed"
    REVIEW_COMPLETED = "review_completed"
    RECONCILIATION_RUN = "reconciliation_run"
    DRIFT_DETECTED = "drift_detected"
    ARTIFACT_READ = "artifact_read"
    ARTIFACT_WRITTEN = "artifact_written"

  class Event(BaseModel):
    kind: EventKind
    timestamp: datetime
    artifact_id: str
    detail: dict[str, Any]  # event-specific payload

  v1 emitter: write to a JSONL file or Python logging. Future: Unix socket for
  spec-driver TUI integration (spec-driver already has artifact_event.py doing
  similar for file touches).

  Reconciliation

  Reconciliation is observation + comparison, not mutation.

  class DriftKind(StrEnum):
    SESSION_MISSING = "session_missing"      # artifact says active, session 
  dead
    SESSION_ORPHANED = "session_orphaned"    # session alive, no artifact 
  reference
    STATUS_MISMATCH = "status_mismatch"      # artifact status != session 
  reality
    STALE_REVIEW = "stale_review"            # review-index stale vs current 
  head

  class DriftEntry(BaseModel):
    kind: DriftKind
    role: Role | None
    expected: str
    actual: str
    detail: str

  class ReconciliationReport(BaseModel):
    artifact_id: str
    timestamp: datetime
    drift: list[DriftEntry]
    healthy: bool  # len(drift) == 0

  reconcile() returns this report. The caller decides what to do вҖ” re-spawn a
  dead session, kill an orphaned one, re-prime a stale review. autobahn provides
   the diagnosis, not the prescription.

  This aligns with the "artifact state is authoritative" principle вҖ”
  reconciliation detects where runtime reality has diverged from what artifacts
  say should be true, and reports it.

      724 +

      727 +### Three sandbox layers exist in practice                    
      728 +

      730 +- Claude Code uses bwrap / sandbox-exec internally            
      731 +- Codex uses its own sandbox-exec integration

      746 +

  Adversarial Review of DR-001

  Attacking the design for imprecision, flawed reasoning, hidden assumptions,
  weak verification, and missing detail.

  1. TransitionPlan.initial_prompt вҖ” who writes this?

  The DR says transition_from_handoff() produces a TransitionPlan with an
  initial_prompt field. But who generates this prompt string? The function takes
   a WorkflowContext (pure data) and returns a plan. Is autobahn constructing
  prose from workflow state?

  Problem: This flirts with the brief's explicit anti-pattern: "should not be
  designed around freeform prompt construction as the primary abstraction." If
  autobahn is templating agent prompts, it's doing prompt engineering вҖ” that's
  arguably spec-driver's or the harness's job.

  Resolution options:
  - A. initial_prompt is a reference/pointer (e.g. "resume from handoff at
  workflow/handoff.current.yaml"), not generated prose
  - B. Prompt templates live in spec-driver (installed as agent config),
  autobahn just fills in variables (artifact_id, phase_id, role)
  - C. Remove initial_prompt from TransitionPlan; harness adapter reads the
  handoff directly and constructs its own launch args

  Recommendation: A or B. The prompt should be minimal and mechanical вҖ” "review
  DE-001 phase IP-001.PHASE-03 per workflow/handoff.current.yaml" вҖ” not a
  crafted instruction. The agent's spec-driver skills handle the rest once it
  boots.

  2. OperationResult вҖ” exception or result type?

  The DR uses OperationResult[T] as a generic result wrapper. But the async
  session functions could also raise exceptions. The design doesn't clarify when
   to use which вҖ” are exceptions reserved for programming errors, and result
  types for expected failures? Or is everything wrapped?

  Problem: Mixed error handling strategies are a common source of bugs. Callers
  won't know whether to try/except or check .success.

  Resolution: Clarify the convention. Suggest: exceptions for programming errors
   and invariant violations (wrong schema version, missing required file).
  OperationResult for operations that can fail in expected ways (session already
   dead, review not yet complete). Document this in the API module.

  3. Artifact model provisional status вҖ” how provisional?

  The DR says models are "provisional" pending DE-107. But they'll have tests
  written against them, code depending on them, and callers importing them.
  "Provisional" doesn't mean "replaceable" once they're load-bearing.

  Problem: The migration path from own-models to imported-models is
  underspecified. Will the pydantic models have the same field names? Same
  nesting? Same validation behaviour?

  Resolution: This is acceptable as a conscious assumption. The models are
  structurally compatible by design. When DE-107 lands, the migration is:
  replace model classes, keep the same field access patterns. If spec-driver's
  models diverge structurally, that's a new DR. Note this explicitly.

  4. sessions.yaml write вҖ” concurrent access?

  autobahn writes sessions.yaml atomically. But what if two autobahn invocations
   run concurrently against the same artifact? Or if an agent also reads
  sessions.yaml while autobahn is writing?

  Problem: Atomic rename prevents corruption but doesn't prevent lost updates.
  Two concurrent writers will race.

  Resolution: Acceptable for v1 вҖ” the design assumes single-orchestrator. If
  multi-client coordination is needed later, that's explicitly deferred. Note
  the assumption.

  5. verify_review_outcome depends on CLI that doesn't exist

  The vertical slice includes verify_review_outcome() which reads files written
  by spec-driver review CLI commands. Those commands don't exist yet.

  Problem: The vertical slice can't run end-to-end until spec-driver builds the
  review CLI.

  Resolution: This is already noted in OQ-001/OQ-003. For testing, use fixture
  files that match the expected schema. The vertical slice proves autobahn's
  orchestration logic; the spec-driver CLI dependency is a sequencing
  constraint, not a design flaw.

  6. Weak: harness executable path

  RuntimePolicy.harness_executable defaults to None, meaning the adapter picks a
   default (e.g. "claude"). But the three-layer model says the caller provides
  the executable. Who actually decides?

  Problem: Two places to specify the command вҖ” policy and adapter default.
  Precedence isn't explicit.

  Resolution: Policy overrides adapter default. Make this explicit: adapter
  provides default command, RuntimePolicy.harness_executable overrides if set.

  7. Missing: how does load_context find the artifact?

  load_context(artifact_dir) takes a Path. But in practice, the caller says
  "load DE-001" not "load /path/to/.spec-driver/deltas/DE-001-slug/". Who
  resolves artifact ID to path?

  Problem: The DR doesn't address artifact discovery/resolution.

  Resolution: This is intentionally outside load_context вҖ” the caller resolves
  the path. Could be a utility function, could be spec-driver CLI (spec-driver 
  show delta DE-001 --path). Don't embed discovery in the API function.

  8. Doctrinal alignment check

  - "No implementation without a spec-driver artefact" вҖ” satisfied, DE-001
  exists
  - Ceremony mode town_planner вҖ” delta-first flow followed
  - Commit policy вҖ” .spec-driver/** committed promptly
  - Adapter isolation вҖ” coupling rules explicit
  - Artifact-first вҖ” core principle throughout

  No doctrinal conflicts found.
